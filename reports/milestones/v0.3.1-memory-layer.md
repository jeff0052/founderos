# v0.3.1 — Memory Layer v1 三层记忆模型

**日期：** 2026-03-18
**Commit：** 93e8633
**状态：** ⚠️ 代码完成，MCP Tools 未接入路由

---

## 背景

FounderOS 的 FPMS 能管项目，CTO Agent 能执行，但系统缺少"记忆"。每次新 session，Agent 从零开始，靠读文件重建 context——慢、不完整、无法积累。Knowledge System v1 需求文档（1066 行）定义了完整蓝图，Memory Layer 是 Phase 0 的核心交付。

## 目标

建立结构化记忆底座：
- 三层模型（工作记忆 → 短期 → 长期/常驻）
- 事件溯源（所有变更可追溯）
- 记忆衰减 + 自动晋升机制
- 为 Phase 1（信号检测、关系图谱）提供存储基础

## 交付物

- **spine/memory.py** — 350 行核心逻辑
  - MemoryStore 类：add, search, update, forget, promote, confirm, decay
  - 三层模型：working → short_term → long_term → resident
  - 事件溯源：memory_events 表，append-only
  - 相似度检测：防重复写入
  - 常驻记忆：bootstrap 时自动加载到 context
- **6 个 MCP Tools 定义**（memory_add/search/update/forget/promote/confirm）
- **68 个新测试** → 总计 576 tests
- **22 条记忆入库，3 条进入常驻层**
- **PRD 四轮迭代** v1 → v1.1 → v2 → v2.1 → v2.2（含 GPT 交叉审查）

## 关键决策

1. **三层而非五层** — 需求文档设计了五层，实现时简化为三层+常驻，够用且不过度设计
2. **事件溯源** — 所有写入 append-only，可审计可回溯
3. **bigram 相似度** — 简单有效的去重方案，不依赖外部向量库
4. **PRD 交叉审查** — 用 GPT 审查 Claude 写的 PRD，多模型交叉验证
