## 背景

FounderOS 从 v0.1.0 到 v0.4.0 四天内交付了 FPMS、CTO Agent、Constitution Guard、Memory Layer，但开发流程每次都不一样。主 Agent 经常自己干活而不是派 CTO、用 TODO.md 而不是 FPMS、做完不更新全局文档导致 session 断裂。Jeff 无法监督方向，换 session 就得从头解释。

## 目标

1. 固定开发流程，不再临场发挥
2. 解决 session 间上下文断裂
3. 给 CTO Agent 装上标准化工具链

## 交付物

### 三层文档体系
- **FPMS** — what（任务实时状态，机器 source of truth）
- **OVERVIEW.md** — where（系统架构和方向，Jeff 的监督窗口）
- **MEMORY.md** — how（工作约定和教训，Agent 的连续性）
- 三者不重叠，各司其职

### 开发 SOP（总分总循环）
- **设计阶段**：Jeff + 主Agent 充分讨论，不动手。任务拆到"一个 task 一个 session 能做好"的颗粒度，每个 task 有验收标准
- **执行阶段**：CTO Agent 按方案逐个执行，按验收标准验收，测试完整
- **收尾阶段**：开发报告 + 更新 OVERVIEW/MEMORY + 写文档 + commit
- **铁律：没有"回到总"这一步，任务不算完成**

### gstack 工具链（给 CTO Agent）
- 安装 gstack (garrytan/gstack) 到 `.claude/skills/`
- 13 个 skill 已注册
- Browse binary + Playwright Chromium 就绪
- `.claude/CLAUDE.md` 配置 CTO 启动流程
- 现阶段启用：/plan-eng-review, /review, /ship, /retro
- 未来启用：/qa, /browse, /design-review（有 UI 时）

### 角色分工
- **Jeff** — CEO，确认方向和方案
- **主 Agent** — 协调者，讨论需求、拆任务、派活、验收
- **CTO Agent** — 执行者，用 gstack 干活

## 关键决策

1. **用 gstack 的流程骨架，不重复造轮子** — 参考 Garry Tan 的方法论
2. **整套装上分阶段启用** — 以后做 UI 产品不用重新搞流程
3. **讨论阶段不动手** — Jeff 说"讨论充分再做"，这是铁律
4. **/plan-ceo-review 由 Jeff 本人替代** — 他就是 CEO，不用 AI 演

## 踩过的坑

- 主 Agent 不自觉地越俎代庖，直接写文件 commit 而不是先讨论
- 写了 SOP 说"Jeff 确认后才执行"，自己第一个就违反了
- 讨论阶段 ≠ 执行阶段，必须严格分开
