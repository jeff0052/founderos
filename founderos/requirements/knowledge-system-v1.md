# FounderOS 企业级知识系统需求文档 v1

> 起源：2026-03-13 凌晨对话，从三省治理漏洞 → 技术瓶颈 → 组织架构 → 信息架构的系统思考
> 本质：设计一个企业级的大脑记忆系统，不是文件管理工具

---

## 1. 问题的本质

### 1.1 我们在解决什么问题

一个极小团队（创始人+AI）要运营三条跨国支付业务线，面对的信息处理挑战：

**信息输入端：**
- 每天数十封商务邮件（合作伙伴、监管机构、投资人、团队）
- 每周数次关键会议（NetStar日本、Paymongo菲律宾、PDAX加密、各国渠道）
- 持续的市场变化（监管政策、竞品动态、汇率波动、技术趋势）
- 内部运营数据（项目进展、资金状况、技术债务、团队状态）
- 社交信号（Twitter/LinkedIn上的行业动态、关键人物发言）

**信息处理需求：**
- 从海量信息中识别关键signal
- 维持对所有活跃项目的准确认知
- 追踪数十个并行关系的状态和温度
- 在任意时刻做出有据可依的决策
- 积累可复用的经验和判断模式

**信息输出端：**
- 精准的商务沟通（邮件回复、合同条款、谈判策略）
- 及时的决策支持（该push还是wait、该投入还是放弃）
- 系统化的执行跟踪（谁在做什么、什么blocked、下一步是什么）
- 周期性的战略复盘（方向对不对、资源分配合理吗）

### 1.2 为什么现有方案不够

**当前架构：**
```
workspace/
├── MEMORY.md (共享核心记忆, <2KB限制)
├── memory/
│   ├── YYYY-MM-DD.md (每日流水记录)
│   ├── topics/ (主题文档)
│   ├── lessons/ (踩坑记录)
│   ├── decisions/ (治理决策)
│   ├── monthly/ (空)
│   └── weekly/ (空)
├── founderos/
│   ├── vision.md, strategy.md, governance.md
│   ├── methodology.md (CDRE)
│   ├── missions/ (onta.md, uniweb.md, license.md)
│   └── dashboard.html
└── skills/ (各种AI技能)
```

**当前架构的根本缺陷：**

| 缺陷 | 具体表现 | 根因 |
|------|----------|------|
| 记忆容量有限 | MEMORY.md限制2KB，每日文件几KB | 为节省token设计，牺牲了信息完整性 |
| 信息扁平化 | 所有信息按日期堆叠，无语义关联 | 缺乏概念层级和关系网络 |
| 被动式记忆 | AI只记录Jeff主动告知的内容 | 缺乏主动采集和监控能力 |
| 无抽象机制 | 细节和概念混在一起 | 缺乏从细节到概念的蒸馏流程 |
| 单一视角 | 没有基于愿景的signal过滤 | 缺乏统一的观察者框架 |
| 不可扩展 | 文件多了检索效率指数下降 | 纯文件系统，无索引无搜索 |
| 知识断层 | 隐性知识无法系统化捕获 | 缺乏从实战到知识的转化机制 |
| Context碎片 | 每次新session要重建全部context | 缺乏高效的context恢复机制 |

### 1.3 组织现状详细画像

**三条业务线的信息复杂度：**

**Uniweb（法币世界）：**

**跨业务线的共享层：**
- 底层基础设施：牌照、结算引擎、合规系统、支付通道

## 2. 设计哲学（深入展开）

### 2.1 观察者框架理论

**哲学基础：**
佛家"分别心"的概念——一切数据本身无意义，意义来自观察者的框架。企业的愿景就是组织的"分别心"，决定了：
- 什么是signal（与使命相关的变化）
- 什么是noise（与使命无关的干扰）
- 什么是opportunity（推进使命的可能性）
- 什么是threat（阻碍使命的风险）

**FounderOS的观察者框架：**
```
核心使命：让支付基础设施民主化
         ↓
三个观察维度：
├── 法币维度：传统金融基础设施的可达性
├── Web3维度：去中心化支付的普及度
└── AI维度：智能支付的自动化程度
         ↓
判断标准：这个信息是否影响以上三个维度的进展？
├── 正向影响 → signal (opportunity)
├── 负向影响 → signal (threat)
├── 无影响 → noise
└── 不确定 → 标记观察，积累数据后再判断
```

**观察者框架的动态性：**
- 框架本身不是静态的，随着业务发展会演化
- 早期：关注市场准入（能不能做）
- 中期：关注市场扩张（怎么做大）
- 后期：关注市场防御（怎么保持优势）
- 每个阶段的signal识别标准不同

**实际应用示例：**

| 信息 | 无框架的判断 | 有框架的判断 |
|------|------------|------------|
| "日本修改稳定币法规" | 可能重要？ | 🚨生存级：直接影响NetStar合作和日本市场进入策略 |
| "Stripe发布新API" | 技术新闻 | 🟡机会级：参考其架构设计，评估是否影响我们的技术选型 |
| "某influencer推荐加密钱包" | 市场热点 | ⚪noise：与我们的B2B支付基础设施使命无关 |
| "菲律宾央行发布新通告" | 监管新闻 | 🚨生存级：直接影响Onta菲律宾的合规框架和运营牌照 |
| "OpenAI发布GPT-5" | AI大新闻 | 🟡机会级：评估对Vibecash产品能力的影响 |
| "全球芯片短缺加剧" | 供应链新闻 | ⚪noise：与支付基础设施无直接关系 |

### 2.2 生物进化模型（深入展开）

**人眼光谱进化的启示：**
- 人眼能看到的光谱（380-700nm）不是"最好的"，是"最适应地球环境的"
- 蛇能看红外线（捕猎需要），蜜蜂能看紫外线（采蜜需要）
- 每种生物的"观察能力"都是环境压力下的最优适应
- **FounderOS的"视觉系统"也应该是环境压力下进化的，而不是预先设计的**

**进化机制在系统中的应用：**

```
变异（Variation）：
└── 系统尝试不同的信息分类方式和signal识别规则

选择（Selection）：
└── 有效的分类被保留（帮助做出好决策的）
└── 无效的分类被淘汰（导致错误判断或浪费注意力的）

遗传（Inheritance）：
└── 有效规则被写入长期记忆，影响未来的判断

适应（Adaptation）：
└── 整个系统随业务环境变化不断调整
```

**实际例子：**
- 第1个月：系统把所有监管新闻标记为🚨，Jeff发现大部分不相关 → 进化：细分监管新闻为"我们运营国家的"和"其他国家的"
- 第3个月：发现"其他国家的"监管变化中有些预示全球趋势 → 进化：增加"趋势预警"分类
- 第6个月：积累足够数据后发现某些指标是领先指标 → 进化：建立预测模型

### 2.3 人脑记忆模型（深入展开）

**人类记忆系统的四层架构：**

```
感觉记忆（Sensory Memory）- 持续<1秒
├── 所有感官输入的瞬间缓存
├── 绝大部分被丢弃
├── 只有被注意力选中的进入下一层
└── 对应系统：原始数据输入流（邮件、消息、市场数据）

短期/工作记忆（Working Memory）- 持续秒-分钟，容量7±2
├── 当前正在处理的信息
├── 极其有限的容量
├── 不编码就会消失
└── 对应系统：AI的Context Window（当前对话中活跃的信息）

长期记忆-陈述性（Declarative Long-term）- 持续天-年
├── 情节记忆：具体事件（"3月12日和吉田开会"）
├── 语义记忆：抽象知识（"日本稳定币监管要求"）
├── 需要编码和巩固过程
└── 对应系统：结构化文档库（会议记录、知识库、决策档案）

长期记忆-程序性（Procedural Long-term）- 持续终身
├── 技能和模式（"怎么和日本商务伙伴沟通"）
├── 内隐的，难以言语化
├── 通过反复实践形成
└── 对应系统：隐性知识库（判断框架、沟通模式、决策模板）
```

**人脑的关键机制和系统对应：**

**注意力过滤机制（Attention Filter）：**
- 人脑每秒接收1100万比特感官信息，但只能有意识处理50比特
- 过滤率：99.99955%
- 过滤依据：与当前目标的相关性、新奇度、情绪显著性
- **系统对应**：Signal Detection Engine必须实现类似的过滤率
- 在几千万文件中，任一时刻只有极少量信息进入工作记忆

**记忆巩固机制（Memory Consolidation）：**
- 人脑在睡眠期间将重要的短期记忆转化为长期记忆
- 不是简单复制，而是抽象提炼（提取要点，丢弃细节）
- 与已有知识网络建立关联
- **系统对应**：定期概念蒸馏流程
  - 原始会议记录 → 提取关键决策和signal → 更新概念图谱
  - 多次类似经验 → 抽象为判断模式 → 写入程序性记忆

**遗忘曲线机制（Forgetting Curve）：**
- 艾宾浩斯遗忘曲线：不复习的信息快速遗忘
- 但遗忘不是bug，是feature——帮助大脑释放容量给更重要的信息
- 被多次recall的信息记忆更牢固
- **系统对应**：信息衰减和强化机制
  - 长期未被引用的信息自动降低检索权重
  - 被频繁引用的信息提升权重并可能升级到更高抽象层
  - 过时的信息不删除但归档，仅在深度检索时出现

**联想网络机制（Associative Network）：**
- 人脑中的概念不是存在文件夹里，而是通过联想链接组织
- "NetStar" → 日本 → 吉田 → API → 4月8日邀请 → PR release
- 激活一个概念会自动激活相关概念（扩散激活 spreading activation）
- **系统对应**：概念图谱 + 语义检索
  - 查询"NetStar"自动带出所有关联实体和事件
  - 按关联强度排序，最强关联优先出现
  - 支持多跳推理："NetStar" → "日本监管" → "稳定币法规变化" → "影响我们什么"

**元认知机制（Metacognition）：**
- 人能意识到自己"知道什么"和"不知道什么"
- "我知道NetStar的API状态，但不确定日本最新的监管要求"
- 这种自我评估能力对决策质量至关重要
- **系统对应**：知识置信度标记
  - 每个知识点标记置信度（confirmed / likely / uncertain / outdated）
  - 基于信息来源、时效性、交叉验证自动评估
  - 决策支持时明确标注"这些是确定的，这些需要验证"

### 2.4 分形结构理论（深入展开）

**自然界的分形：**
- 树：树干 → 树枝 → 小枝 → 细枝，每层结构相似
- 肺：气管 → 支气管 → 细支气管 → 肺泡，23级分支
- 血管：动脉 → 小动脉 → 微动脉 → 毛细血管
- 河流：干流 → 支流 → 溪流 → 细流

**分形的核心优势：**
- 用简单规则生成复杂结构
- 每层都可以用相同的方式理解和处理
- 天然支持无限扩展
- 局部和整体具有自相似性

**知识系统的分形设计：**

```
最小单元（细胞）= 一条知识记录
├── Header: 时间、来源、类型、关联实体、Signal级别、置信度
├── Summary: 一句话概要（<50字）
├── Body: 完整内容
├── Links: 关联到其他知识记录的引用
└── Meta: 访问频率、最后引用时间、衰减权重

组织单元（组织）= 一个主题/项目/关系
├── Header: 主题名、状态、优先级、负责人、健康度
├── Summary: 当前状态概要（<200字）
├── Records: 包含的所有知识记录（按时间/重要性排序）
├── Links: 关联到其他主题/项目/关系
└── Meta: 活跃度、最近更新、trend方向

业务单元（器官）= 一条业务线
├── Header: 业务线名、战略定位、关键指标、当前阶段
├── Summary: 业务线整体状态（<500字）
├── Themes: 包含的所有主题/项目/关系
├── Links: 与其他业务线的依赖和协同
└── Meta: 健康度评分、增长趋势、风险等级

企业单元（生命体）= FounderOS整体
├── Header: 使命、愿景、当前战略阶段
├── Summary: 企业整体状态（<1000字）
├── Business Lines: 三条业务线
├── Cross-cutting: 共享基础设施、战略决策、资源分配
└── Meta: 整体健康度、战略对齐度、外部环境评估
```

**每层遵循相同的操作接口：**
- `getSummary()` — 获取该层级的概要（用于上层理解）
- `getDetails()` — 展开该层级的详细内容（用于深入分析）
- `getSignals()` — 获取该层级的活跃signal
- `getHealth()` — 获取该层级的健康度评估
- `getLinks()` — 获取该层级的关联关系
- `update(record)` — 添加新的知识记录
- `distill()` — 从下层细节蒸馏出本层概要

### 2.5 硅基智能最大化原则

**碳基（人类）vs 硅基（AI）的能力矩阵：**

| 能力维度 | 碳基优势 | 硅基优势 |
|----------|----------|----------|
| 模式识别 | 直觉、第六感、微表情 | 大规模数据中的统计模式 |
| 记忆 | 情感记忆、肌肉记忆 | 海量精确存储、完美recall |
| 推理 | 常识推理、类比思维 | 逻辑推理、多维分析 |
| 创造 | 跨域灵感、审美判断 | 组合创新、变体生成 |
| 沟通 | 情商、语境理解、信任 | 多语言、24/7、一致性 |
| 决策 | 不确定性下的勇气 | 数据驱动的概率评估 |
| 学习 | 从少样本中学习、迁移 | 从大量数据中提取模式 |
| 注意力 | 灵活聚焦、优先级直觉 | 并行监控、不遗漏 |

**系统设计原则：**
- **让Jeff做Jeff擅长的事**：关系建立、直觉判断、战略决策、谈判
- **让AI做AI擅长的事**：信息监控、模式分析、记忆维护、执行跟踪
- **接口设计**：Jeff输入高维判断（"这个感觉不对"），AI负责结构化和追踪
- **反馈循环**：AI的分析结果反馈给Jeff，Jeff的判断修正AI的模型

## 3. 系统架构需求

### 3.1 信息采集层（感觉器官）

#### 3.1.1 邮件采集系统
**需求描述：**
Jeff将CC所有商务邮件给系统，系统自动读取、分类、提取关键信息。

**详细需求：**
- 接收邮件并解析：发件人、收件人、主题、正文、附件
- 自动识别邮件类型：商务洽谈、合同文件、内部沟通、信息通知、垃圾邮件
- 提取关键实体：人名、公司名、金额、日期、承诺事项
- 建立邮件线索（thread）追踪：同一主题的邮件串联
- 识别行动项（action items）：谁需要做什么，什么时候
- 检测情绪和紧急度：对方是否在催促、是否有不满信号
- 自动关联到已知的合作伙伴/项目

**示例场景：**
```
输入：来自 yoshida@netstar.jp 的邮件
"Dear Jeff, please find attached the API documentation as discussed. 
We look forward to starting the technical integration. 
Also, please confirm your availability for the April 8th PR event."

系统自动处理：
├── 识别：NetStar / 吉田 / API文档交付 / 技术对接启动
├── Signal级别：🟡机会级（技术对接进展，非紧急但重要）
├── Action items：
│   ├── 下载并转发API文档给技术团队
│   └── 确认4月8日日本行程
├── 关联更新：
│   ├── NetStar合作状态：API → 已获取 ✅
│   ├── 日本市场进入：技术对接 → 进行中
│   └── 日程：4月8日 → 日本PR event（待确认）
└── 存储：partnerships/netstar/emails/2026-03-12-api-handover.md
```

#### 3.1.2 会议记录系统
**需求描述：**
会议后自动或半自动生成结构化记录，提取关键决策和行动项。

**详细需求：**
- 支持多种输入方式：
  - 语音转录（如果会议有录音）
  - Jeff口头总结（通过Telegram语音消息）
  - 文字简述（Jeff发几句关键点）

### 3.2 信息处理层（大脑皮层）

#### 3.2.1 Signal Detection Engine（信号检测引擎）

**核心功能：**
基于观察者框架（支付民主化使命），对所有进入系统的信息进行分级和路由。

**Signal分级详细定义：**

```
🚨 CRITICAL — 生存级Signal
├── 定义：直接影响公司存亡或核心业务线的事件
├── 响应时间：立即（<1小时）
├── 通知方式：直接push给Jeff（Telegram消息 + 必要时电话提醒）
├── 处理流程：立即分析影响 → 提出应对方案 → 等待Jeff决策
├── 示例：
│   ├── 核心合作伙伴中断合作
│   ├── 运营国家禁止我们的业务类型
│   ├── 资金链断裂风险
│   ├── 关键人员离职
│   └── 安全事故/数据泄露

🟡 IMPORTANT — 机会/威胁级Signal  
├── 定义：影响增长速度、竞争力或战略方向的事件
├── 响应时间：当日内
├── 通知方式：包含在每日briefing中，重要的单独通知
├── 处理流程：评估影响范围 → 分析选项 → 建议行动 → Jeff决策
├── 示例：
│   ├── 竞品获得大额融资
│   ├── 新市场出现进入窗口
│   ├── 合作伙伴态度变化
│   ├── 技术架构需要重大调整
│   └── 团队能力缺口显现

🟢 TREND — 趋势级Signal
├── 定义：长期影响战略方向的趋势性变化
├── 响应时间：周级别
├── 通知方式：包含在周度复盘中
├── 处理流程：记录趋势 → 关联已有pattern → 更新战略模型
├── 示例：
│   ├── 行业技术标准演变
│   ├── 全球监管趋势变化
│   ├── 用户行为模式变化
│   ├── 新技术成熟度变化
│   └── 宏观经济周期信号

⚪ NOISE — 噪音
├── 定义：与使命无直接关系的信息
├── 响应时间：不响应
├── 处理方式：自动过滤，不进入工作记忆
├── 但保留原始数据以备未来重新评估（context变化后noise可能变signal）
```

**Signal识别的技术实现需求：**
- 基于规则的初筛：关键词匹配、来源过滤、实体识别
- 基于语义的精筛：理解信息含义，判断与使命的关联度
- 基于历史的校准：参考过去类似信息的处理结果
- 人工校准接口：Jeff可以纠正分级错误，系统从纠正中学习

#### 3.2.2 概念蒸馏引擎（Concept Distillation Engine）

**核心功能：**
定期将底层细节信息蒸馏为高层抽象概念，维持系统的认知效率。

**蒸馏的层级和频率：**

```
Level 0: 原始数据 → Level 1: 事件记录（实时）
├── 邮件原文 → "NetStar吉田发送了API文档"
├── 会议录音 → 结构化会议记录
└── 新闻文章 → 关键事实提取

Level 1: 事件记录 → Level 2: 状态更新（每日）
├── 多封NetStar邮件 → "NetStar合作状态：技术对接进行中，MOU待签"
├── 多次内部讨论 → "发卡项目状态：blocked on 技术架构"
└── 多条市场新闻 → "日本稳定币监管趋势：开放中，4月可能有新规"

Level 2: 状态更新 → Level 3: 概念总结（每周）
├── NetStar + Gpay + Benji状态 → "亚洲渠道扩张：日本就绪、越南临门、菲律宾布局中"
├── 所有技术blocked → "技术瓶颈：3个项目等内部开发，CTO角色急需"
└── 所有监管动态 → "合规环境：6个运营国家中4个明确支持，2个观望"

Level 3: 概念总结 → Level 4: 战略洞察（每月）
├── 渠道 + 技术 + 合规 → "战略评估：市场窗口打开，执行力是唯一瓶颈"
├── 跨业务线对比 → "资源重分配建议：Onta优先，Uniweb稳步，Vibecash等待"
└── 外部环境综合 → "行业判断：2026是稳定币支付的爆发年"
```

**蒸馏质量保证：**
- 每次蒸馏保留对原始数据的引用链接（可追溯）
- Jeff定期抽查蒸馏结果，纠正偏差
- 关键决策必须能追溯到原始数据
- 蒸馏不等于删除：原始数据永远保留，只是降低检索优先级

#### 3.2.3 关系图谱引擎（Relationship Graph Engine）

**核心功能：**
维护所有外部和内部实体的关系网络，支持关联查询和推理。

**实体类型：**
```
人物（Person）
├── 属性：姓名、公司、角色、联系方式、沟通风格
├── 关系：属于哪个公司、参与哪些项目、认识哪些其他人
└── 动态属性：最近互动时间、关系温度、可靠性评估

组织（Organization）
├── 属性：名称、类型、规模、业务范围、所在地
├── 关系：合作伙伴、竞品、潜在目标、投资方
└── 动态属性：合作状态、战略重要性、风险等级

项目（Project）
├── 属性：名称、业务线、阶段、优先级、负责人
├── 关系：涉及哪些组织和人物、依赖哪些其他项目
└── 动态属性：进展百分比、健康度、blocked原因

事件（Event）
├── 属性：时间、类型、参与方、内容
├── 关系：属于哪个项目、涉及哪些人物和组织
└── 动态属性：后续行动状态、影响评估

Signal
├── 属性：时间、来源、内容、级别、类型
├── 关系：影响哪些项目/组织/人物
└── 动态属性：处理状态、实际影响（回顾评估）
```

**关系类型和强度：**
```
合作关系：强度（深度合作/浅层接触/潜在）、温度（热/温/冷）、历史
竞争关系：直接/间接、威胁程度、差异化点
依赖关系：方向（我方依赖/对方依赖/双向）、关键度、替代方案
影响关系：谁能影响谁的决策、影响力大小
```

**自动健康检测：**
- 关系冷却预警：超过N天没有互动的合作关系自动提醒
- 风险信号：对方回复变慢、语气变化、承诺未兑现
- 机会信号：对方主动接触增加、新的合作可能性出现
- 关系网络效应：A和B都认识C，是否可以三方联动？

### 3.4 检索与注意力层（前额叶皮层）

#### 3.4.1 语义检索系统

**需求描述：**
支持基于语义（而非关键词）的信息检索，在几千万文件中快速找到相关信息。

**技术需求：**
- **向量化存储**：每个文档的summary和body转为语义向量
- **多维索引**：支持按时间、实体、主题、signal级别等多维度检索
- **相关性排序**：综合考虑语义相似度、时效性、重要性、访问频率
- **上下文感知**：检索结果根据当前对话context动态调整
- **模糊检索**：支持不精确查询（"之前和日本那个公司聊的事情"）
- **跨文档推理**：能从多个文档中综合信息回答复杂查询

**性能需求：**
- 检索延迟：<2秒（日常查询），<10秒（复杂跨文档查询）
- 准确率：top-5结果中包含正确答案的概率 >90%
- 规模：支持百万级文档without性能下降

**注意力分配规则：**
```

相关优先：与当前对话主题直接相关的信息优先
新鲜优先：最近更新的信息优先于旧信息
频繁优先：被频繁引用的信息优先（说明重要性高）
置信优先：confirmed的信息优先于uncertain的
```

**注意力切换机制：**
- 检测到话题切换：自动替换L2/L3 context
- 检测到紧急signal：中断当前处理，优先处理signal
- 检测到context过载（>100K）：主动提醒Jeff开新session

### 3.5 进化与学习层（小脑 + 基底神经节）

#### 3.5.1 系统自优化

**需求描述：**
系统根据使用反馈不断优化自身的信息处理能力。

**自优化维度：**
- **Signal识别准确率**：
  - 追踪Jeff对signal分级的纠正
  - 分析纠正模式，调整识别规则
  - 目标：每月识别准确率提升5%
  
- **信息检索效率**：
  - 追踪Jeff对检索结果的满意度（是否需要多次检索才找到）
  - 优化索引策略和排序算法
  - 目标：first-hit准确率持续提升

- **概念蒸馏质量**：
  - Jeff对蒸馏结果的纠正频率
  - 蒸馏后的概念是否在后续决策中被有效使用
  - 目标：蒸馏准确率>85%

- **预测能力**：
  - 系统预测的下一步行动是否与Jeff的决策一致
  - 预测的signal是否确实影响了业务
  - 目标：从reactive向proactive演进

#### 3.5.2 知识衰减与更新

**需求描述：**
管理知识的时效性，确保系统不会被过时信息误导。

**衰减机制：**
```
每个知识条目有 decay_weight（0.0-1.0）：
├── 初始值：1.0
├── 每日衰减：weight *= 0.999（半衰期约2年）
├── 被引用时：weight = min(1.0, weight + 0.1)（使用增强）
├── 被Jeff纠正为过时：weight = 0.1（快速降权）
├── 关联实体发生重大变化：weight *= 0.5（关联降权）
└── 检索排序时使用weight作为因子之一
```

**更新触发条件：**
- 新信息与现有知识冲突 → 触发验证流程
- 知识条目的decay_weight低于阈值 → 标记为需要更新
- 关联实体的状态发生变化 → 检查所有关联知识是否需要更新
- 定期审计：每月检查所有P0知识条目是否仍然准确

## 4. 约束条件（详细）

### 4.1 资源约束

**Token成本预算：**
```
每日预算分配：
├── 日常对话（主assistant）：~50K tokens/天
├── Signal检测和处理：~10K tokens/天
├── 概念蒸馏（每日）：~5K tokens/天
├── 概念蒸馏（每周）：~20K tokens/周
├── 专业角色（按需）：~20K tokens/次
├── 外部情报采集：~10K tokens/天
└── 总计：~80-100K tokens/天（日常）

成本优化策略：
├── 分层加载：90%对话只需<5K tokens的context
├── 缓存机制：频繁使用的概念缓存，不重复加载
├── 批量处理：Signal检测和蒸馏集中在低峰时段
└── 按需升级：日常用Sonnet，复杂分析切Opus
```

**基础设施约束：**
- 当前：Mac mini + OpenClaw + Telegram
- 存储：本地文件系统 + Git版本控制
- 计算：本地CPU + 云端LLM API
- 网络：标准互联网连接
- 未来扩展需求：可能需要向量数据库、图数据库、定时任务服务

### 4.2 设计约束

**分形原则执行标准：**
- 任何新的存储结构必须遵循"Header-Summary-Body-Links-Meta"模式
- 任何新的分类维度必须能在所有层级一致应用
- 系统复杂度增长必须是对数级（O(log n)），不是线性级（O(n)）

**渐进式扩展路径：**
```
Phase 0（现在）：~500个文件
├── 纯markdown文件系统
├── memory_search基本检索
├── 手动分类和关联
└── MEMORY.md + daily notes

Phase 1（1-3月）：~5,000个文件
├── 结构化元数据头
├── 自动索引生成
├── 邮件自动采集
└── 基础Signal Detection

Phase 2（3-6月）：~50,000个文件
├── 向量化语义检索
├── 概念图谱
├── 自动蒸馏流程
└── 多Agent协作

Phase 3（6-12月）：~500,000个文件
├── 完整知识图谱
├── 预测性分析
├── 跨域推理能力
└── 自适应进化

Phase 4（1-3年）：~5,000,000+个文件
├── 企业级搜索引擎
├── 分布式存储
├── AI原生交互界面
└── 组织记忆传承系统
```

**修改弹性要求：**
- 任何Phase的设计决策不能锁死后续Phase的选择
- 数据格式必须可迁移（markdown为基础，元数据为标准YAML）
- 存储引擎可替换（从文件系统平滑迁移到数据库）
- 检索引擎可替换（从文本搜索平滑迁移到向量搜索）

### 4.3 风险管控

**信息安全风险：**
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 商务机密泄露 | 中 | 高 | 访问控制分级、Agent权限隔离、敏感信息标记 |
| 数据丢失 | 低 | 极高 | Git版本控制、定期备份、异地存储 |
| AI幻觉导致错误判断 | 高 | 中 | 置信度标记、关键决策必须人工确认、可追溯性 |
| 单点故障 | 中 | 高 | 核心数据多重备份、降级方案（无AI时人工可用） |

**系统性风险：**
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 概念漂移（蒸馏偏差） | 中 | 中 | 定期人工审计、保留原始数据可追溯 |
| 注意力偏差（重要signal被忽略） | 中 | 高 | 定期回顾missed signals、多维度交叉验证 |
| 过度自动化（AI越权决策） | 中 | 高 | 明确人机分工边界、关键操作需Jeff确认 |
| 复杂度失控 | 高 | 中 | 严格遵循分形原则、定期简化重构 |
| Token成本超标 | 高 | 中 | 预算监控、自动降级策略、定期成本审计 |

**降级方案（当AI不可用时）：**
- 所有核心信息以markdown存储，人类可直接阅读
- 索引文件以人类可理解的格式维护
- 关键决策记录完整，新人可以快速了解背景
- Git历史保留完整变更记录
