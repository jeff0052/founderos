# 分形项目管理系统 — 需求文档

## 1. 背景

### 1.1 AI Agent 的记忆困境

AI Agent 运行在一个根本性的矛盾之中：**它在单次对话内拥有超越人类的认知能力，但在对话之间完全失忆。**

每次 session 启动，Agent 面对的是一张白纸。没有昨天的记忆，没有上周的决策，没有对任何项目的「熟悉感」。它必须从文件中重建对世界的认知——而重建的质量，完全取决于这些文件的组织方式。

这不是一个边缘问题。这是 AI Agent 能否从「对话工具」进化为「持续协作者」的核心瓶颈。

**类比：** 想象你每天早上醒来完全失忆，但你的桌子上放着一叠笔记。你能否有效地继续昨天的工作，完全取决于这些笔记写得多好、组织得多清晰。现在的问题是——这些笔记写得很差。

### 1.2 当前系统的状况与缺陷

**现有的信息存储：**

| 文件 | 用途 | 问题 |
|------|------|------|
| `TODO.md` | 任务清单 | 每个任务仅一行标题，无背景、无进度、无依赖。`- [ ] Charles (Anext): 审核信贷协议` 这行字告诉 Agent 的信息接近于零 |
| `MEMORY.md` | 长期记忆 | 非结构化，随时间增长变成信息垃圾场。重要信息被噪音淹没 |
| `memory/YYYY-MM-DD.md` | 每日日志 | 按时间组织而非按项目组织。要找 Anext 的历史，需要翻阅几十个日期文件 |
| 对话历史 | 当次会话上下文 | Session 结束即丢失，跨 session 不可用 |

**这套系统在任务简单时勉强可用。但当项目变得复杂——多个利益相关方、长时间跨度、相互依赖的决策链——它彻底失败。**

### 1.3 核心问题：Agent 的 Context 管理失效

**一句话定义：Agent 在处理复杂多线程项目时，无法有效管理自身的上下文，导致认知断裂、决策质量下降、执行效率低下。**

这个核心问题拆解为五个子问题：

#### 问题 1：认知断裂 — Agent 不知道「发生了什么」

Agent 醒来后面对一个任务：`审核 Anext 信贷协议`。

它不知道：
- 这个协议已经谈了多久？经历了几个版本？
- 上一版的利率是多少？我们提了什么反对意见？
- Charles 上次回复了什么？我们还在等什么？
- 法律审核进展到哪了？律师什么时候回复的？

**结果：Agent 要么问用户（浪费人类时间），要么猜测（可能犯错），要么从头处理（重复劳动）。**

**真实场景：** Jeff 说"Anext 那边怎么样了"。Agent 翻遍 memory 文件也凑不出完整的时间线。最终给出一个模糊的、可能遗漏关键信息的回答。

#### 问题 2：因果链断裂 — Agent 不知道「为什么」

即使 Agent 知道当前状态（利率 7.8%），它也不知道：
- 为什么从 8.5% 谈到 7.8%？我们的谈判策略是什么？
- 为什么我们选择 Anext 而不是其他银行？
- 为什么担保条款是个问题？具体哪一条有风险？

**没有「为什么」，Agent 无法做出有质量的判断。** 它可能在下一次谈判中提出与之前策略矛盾的建议，因为它根本不记得之前的策略是什么。

**真实场景：** Jeff 问"你觉得 7.8% 能接受吗"。Agent 没有之前的谈判上下文，无法给出有依据的意见。

#### 问题 3：全局视野缺失 — Agent 不知道「还有什么」

Agent 专注于 Anext 时，它看不到：
- Chris 的薪水方案明天要讨论，准备好了吗？
- Paymongo 那边虽然发了 agreement，后续还需要什么？
- 这三个项目之间有没有冲突（比如时间、资源）？
- 有没有什么事情快到 deadline 了但没人在跟？

**结果：Agent 变成「只看眼前」的工具，无法主动提醒、无法统筹安排、无法发现风险。**

**真实场景：** Jeff 花了一小时讨论 Anext，结束后才想起来 Chris 的薪水方案明天要用但还没准备。Agent 完全没有提醒。

#### 问题 4：Context 污染 — 加载了不该加载的信息

为了弥补上述问题，一种天然的反应是「多加载一些」。但这带来新问题：

- 200k 的 context window 看起来大，但加载全部 memory 文件可能就用掉 50k+
- 无关信息不是无害的——它**稀释注意力、降低推理质量**
- Agent 在充满噪音的 context 中做决策，就像人类在嘈杂的环境中思考——效果明显下降

**这是一个两难困境：加载太少 → 遗漏关键信息；加载太多 → 噪音淹没信号。**

**真实场景：** Agent 启动时加载了三个月的 memory 日志（为了不漏掉任何东西），结果 context 用了 80k，留给实际工作的只剩 120k，而且 80k 里 90% 的信息跟当前任务无关。

#### 问题 5：结构僵化 — 系统无法随复杂度增长

当前的 TODO.md 是一维的——扁平的清单。这在管理 5 个独立任务时没问题。但现实中：

- 一个「项目」下面有几十个任务
- 任务之间有依赖关系
- 一些任务属于更大的战略方向
- 复杂度不是线性增长而是指数增长

**扁平清单处理不了树形结构的问题。** 就像用 Excel 管理关系数据库——工具错配。

**真实场景：** TODO.md 里有 30 个任务，分属 5 个不同项目。Agent 无法快速判断"Anext 这个项目整体进展如何"，因为相关任务散落在清单的各个位置，没有归集关系。

### 1.4 问题的影响

| 影响维度 | 表现 | 后果 |
|----------|------|------|
| **人类时间浪费** | Jeff 需要反复向 Agent 解释上下文 | 本应由 Agent 节省的时间反而增加了 |
| **决策质量下降** | Agent 缺乏历史上下文做判断 | 给出的建议可能与之前的策略矛盾 |
| **遗漏风险** | 没有全局视图和主动提醒 | deadline 被错过、blocker 没人跟进 |
| **信任侵蚀** | Agent 反复「忘记」事情 | 用户逐渐失去对 Agent 的信任，退回手动管理 |
| **规模瓶颈** | 项目越多问题越严重 | 系统无法随业务增长而扩展 |

**终极风险：如果 Agent 不能有效管理 context，它就永远只是一个「单次对话的聪明工具」，无法成为真正的「持续协作者」。**

### 1.5 用户

- **主用户：AI Agent** — 需要快速恢复上下文、精准加载信息、跟踪进度、主动提醒、自主维护。系统首先是为 Agent 设计的，因为 Agent 是每天使用它的「人」。
- **次用户：Jeff（人类）** — 需要随时了解全局状态、用自然语言下达指令和查询、review 进展、做关键决策。不需要手动维护任何结构化文件。

## 2. 设计哲学：向人类认知学习

### 2.1 人类 Context 管理机制及启示

| 人类机制 | 原理 | Agent 可学 | 设计启示 |
|----------|------|-----------|----------|
| **分块（Chunking）** | 工作记忆 4-7 块，每块可压缩复杂概念 | ✅ | 每个节点是一个「块」，一句话能恢复认知 |
| **叙事（Narrative）** | 记故事比记清单强 10 倍，时间线+因果链 | ✅ | context 用时间线叙事体，不是无序 bullet |
| **遗忘（Forgetting）** | 自动过滤噪音，重要性加权，是功能不是 bug | ✅ | done 节点定期归档降权，不占 index |
| **周期回顾（Rehearsal）** | 周会/复盘，定期重新加载，对抗遗忘曲线 | ✅ | heartbeat = Agent 的周会 |
| **外化（Externalization）** | 承认记不住，借助笔记/日历/看板 | ✅ | 文件系统就是 Agent 的外化记忆 |
| **关联触发（Association）** | 线索触发记忆链，不是检索而是一条链拉出一串 | ❌ | Agent 做不到，用结构化索引补偿 |
| **社会记忆（Distributed）** | 记忆分布在团队中，不需要一个人记住所有 | ❌ | 目前单 Agent，所有 context 自己扛 |
| **聚焦（Focus）** | 任何时刻只深度关注一件事，其余压入背景 | ✅ | 焦点节点加载完整上下文，其余只保留摘要 |
| **联想打包（Association Bundling）** | focus 时大脑自动拉入相关知识，不需手动检索 | ⚠️ 模拟 | 聚焦某节点时，自动加载关联节点（parent/siblings/依赖），主动搜索 memory 中的相关信息 |
| **视觉分辨率模型（Foveal Vision）** | 注视点高清，周边模糊但不消失，整体画面永远在 | ⚠️ 模拟 | 焦点 = 中央凹（全量上下文），其余 = 周边视觉（一句话摘要），整体画面不丢失 |

### 2.2 Agent 与人类的关键差异

| 维度 | 人类 | Agent |
|------|------|-------|
| Session 内记忆 | 有限（4-7 chunks） | 强（200k window） |
| 跨 Session 记忆 | 有（长期记忆，模糊但持续） | **零（完全失忆）** |
| 检索方式 | 关联触发（自动） | 文件搜索（显式） |
| 遗忘 | 自动、智能 | 要么全记要么全忘 |
| 压缩 | 无意识自动完成 | 需要显式设计 |
| 背景意识 | 模糊但持续存在 | 要么完整加载要么一无所知 |
| 聚焦 | 自动、直觉驱动 | 需要显式机制决定关注什么 |

**核心洞察：**
1. **人类有模糊但持续的背景意识，Agent 要么完整加载要么一无所知。** 系统设计必须弥补这个根本差异。
2. **人类有 focus — 随时知道「现在最该关注什么」。Agent 没有这个直觉，必须用显式机制模拟。**
3. **人类的注意力像眼球：注视点高清，周边模糊，但整体画面永远不丢。** Agent 需要同样的分辨率梯度——不是"加载 or 不加载"的二元选择，而是"高清 / 模糊 / 极模糊"的连续光谱。
4. **人类 focus 时大脑自动联想打包相关知识。** Agent 没有这个能力，必须显式实现：聚焦某个节点时，主动拉入关联信息（parent、siblings、依赖方、memory 中的相关记录），构建一个完整的「认知包」。

### 2.3 硅基 vs 碳基：Agent 的独特优势

| 维度 | 人类（碳基） | Agent（硅基） |
|------|-------------|--------------|
| 回忆精度 | 模糊、会变形、会编造 | 100% 精确，读一次不走样 |
| 结构化处理 | 费劲 | 天然擅长，YAML/JSON 比叙事更快 |
| 并行逻辑 | 极弱 | 可同时处理几十个节点的状态逻辑 |
| 情绪偏差 | 焦虑会扭曲 focus | 可纯粹基于规则 |

**但人类只有 4-7 个 chunk 就能完成极其复杂的任务。** 不是因为记得多，而是因为**每一块都是精准的、高度抽象的。**

外科医生做手术时脑子里不是整本解剖学教科书，而是精确到这一刀需要的那几个关键信息。Agent 应该追求同样的精准度。

### 2.4 设计原则

1. **能耗最优** — 用最少的 context 完成最高质量的任务。加载的每一个 token 都必须服务于当前任务。不服务的 token 就是噪音，噪音不是无害的——它稀释注意力，降低输出质量。
2. **精准胜于全面** — 10k 精准 context > 100k 泛泛 context。Context window 增长的红利应用于更好的抽象和更精准的加载，而不是更多的冗余。
3. **抽象是核心能力** — 节点摘要的质量 = 系统效率的上限。人类 4-7 个 chunk 够用，因为每个 chunk 都是高度抽象的。
4. **分形自相似** — 一个结构，无限嵌套。从原子任务到战略目标，同一 schema。
5. **模式分离** — 概览模式（广度，所有项目一句话）和执行模式（深度，一个任务的完整上下文）不混用。

## 3. 目标

### 3.1 核心目标
1. Agent 能在 **30 秒内**（一次文件读取）恢复全局项目状态
2. Agent 能按需加载任意项目/任务的 **完整上下文**，不浪费 context window
3. 支持从原子任务到战略目标的 **多层级归集**，结构自相似（分形）
4. 任务状态变化时能 **自动向上冒泡**，保持全局视图准确

### 3.2 非目标（v1 不做）
- 不做可视化 dashboard
- 不做多人协作 / 权限管理
- 不做与外部工具（Jira, Notion 等）集成
- 不做自动化工作流引擎

## 4. 功能需求

### FR-1: 统一节点模型（学 Chunking）
- 所有层级（任务 / 里程碑 / 项目 / 战略目标）使用同一数据结构
- 每个节点必须包含：标识、描述、原因、状态、下一步
- 每个节点可选包含：负责人、截止日期、卡点、父节点引用
- 节点通过父节点引用实现层级归集
- **每个节点是一个认知「块」，一句话摘要能让 Agent 恢复对该节点的认知**

### FR-2: 叙事体上下文（学 Narrative）
- 节点的历史记录采用**时间线叙事体**，不是无序列表
- 格式：`日期 + 发生了什么 + 导致什么`，保留因果链
- 叙事体比结构化表格更容易被 Agent 理解和利用
- 示例：
  ```
  3/5 收到初版，利率8.5%太高
  → 3/8 我们回复要求7.2%
  → 3/10 Charles让步到7.8%，但担保条款有问题
  → 3/13 发给律师审核，等意见
  ```

### FR-3: 快速全局概览（补偿 Association 缺失）
- 提供一个紧凑的全局视图文件，按状态分组展示所有节点
- 全局视图控制在 50 行以内，确保一次读取即可掌握全局
- 全局视图需包含：节点标识、一句话描述、当前状态、紧急程度
- **因为 Agent 没有关联触发能力，必须靠结构化索引让 Agent 「看到」所有项目**

### FR-4: 按需深度加载
- 每个节点的详细上下文可以独立存储和加载
- Agent 只在需要时加载特定节点的完整上下文
- 详细上下文包含：完整历史叙事、决策背景、关键联系人、相关文档

### FR-5: 状态管理与冒泡
- 支持状态集：active / waiting / blocked / done / dropped
- 子节点状态变化时，父节点状态需要相应更新
- 归集规则：
  - 任何子节点 blocked → 父节点标记存在 blocked 子项
  - 所有子节点 done → 父节点可标记 done
  - 至少一个 active → 父节点为 active

### FR-6: 智能遗忘与归档（学 Forgetting）
- done 的节点在完成后 **7 天**从 index 移入归档区
- 归档节点不再占用全局视图的 token 预算
- 归档数据保留可查，但不主动加载
- dropped 节点立即从 index 移除，保留在归档中
- **遗忘是功能，不是 bug — 保持 index 干净是对 context window 的尊重**

### FR-7: 快速捕获（Inbox 机制）
- 支持从自然语言对话中快速创建新节点
- 新节点可以先不指定 parent，后续整理时归集
- TODO.md 作为 inbox，Agent 负责定期整理到正式节点中

### FR-8: 主动提醒（学 Rehearsal）
- Heartbeat 时检查：deadline 临近（< 48h）的任务
- Heartbeat 时检查：长期处于 waiting / blocked 的任务（> 7 天）
- 提醒内容包含足够上下文，不需要额外加载
- **Heartbeat = Agent 的周会，是对抗完全失忆的核心机制**

### FR-9: 眼球模型 — 焦点 + 周边视觉 + 联想打包

**设计隐喻：Agent 的注意力应该像人类的眼球。**

注视点（中央凹）看到全部细节；周边视觉模糊但画面不丢；视线移动时新区域 zoom in、旧区域 zoom out。整体画面永远存在。

**三层分辨率：**

| 层级 | 隐喻 | 信息量 | 何时使用 |
|------|------|--------|----------|
| **焦点（Fovea）** | 你正在盯着的地方 | 完整上下文：全量叙事 + 决策历史 + 联系人 + 依赖 | 当前正在处理的 1-3 个节点 |
| **近景（Parafovea）** | 焦点旁边的区域 | 摘要 + 当前状态 + 下一步 + 卡点 | 焦点的 parent、siblings、直接依赖 |
| **背景（Peripheral）** | 视野边缘 | 一句话状态 | 所有其他活跃节点 |

**关键原则：模糊 ≠ 消失。** 背景层的节点虽然只有一句话，但它们**永远存在于全局视图中**。Agent 始终保持对整体画面的感知。

**联想打包机制：**

当 Agent 聚焦到一个节点时，不是只加载这个节点本身，而是自动构建一个**认知包（Context Bundle）**：

1. **向上** — 加载 parent 节点的摘要（理解这个任务属于哪个更大的目标）
2. **平行** — 加载 sibling 节点的状态（理解同一项目下其他任务的进展）
3. **依赖** — 加载被依赖和依赖方的状态（理解什么在等你、你在等什么）
4. **记忆搜索** — 主动在 memory/ 文件中搜索与该节点相关的信息（模拟联想）
5. **时间近邻** — 最近 3 天涉及该节点的对话摘要

**这模拟了人类大脑的联想打包：你想到 Anext，大脑自动拉出 Charles、利率、担保条款、律师、deadline。Agent 必须显式实现这个过程。**

**焦点驱动方式：**
- **用户驱动** — Jeff 提到某个项目/任务，自动聚焦
- **时间驱动** — deadline < 48h 的任务自动升为焦点
- **事件驱动** — 收到相关邮件/消息，自动聚焦对应项目
- **衰减机制** — 超过 3 天没接触的焦点自动降级回近景或背景

**焦点切换（Saccade）：**
- 视线移开时：当前焦点压缩回近景/背景（zoom out），保存状态快照
- 视线移入时：新节点从背景/近景展开为焦点（zoom in），构建认知包
- **切换成本存在但可控** — 保存和加载都是文件操作

**焦点信息记录在全局视图顶部：**
```yaml
focus:
  - id: chris-salary
    reason: 明天12:30要讨论
    since: 2026-03-13
    bundle: [chris-salary, hr-ops, chris-history]
```

**焦点解决的核心问题：Agent 醒来不只是知道「有哪些项目」，而是知道「现在应该关注什么」，并且自动获得关注它所需的全部相关信息。**

### FR-10: 上下文恢复流程
- Agent 新 session 启动时的标准流程：
  1. 读全局视图 → 恢复全局认知 + 知道当前焦点是什么
  2. 自动加载焦点节点的完整上下文
  3. 根据对话主题 → 如需切换焦点，保存旧焦点、加载新焦点
  4. 执行任务 → 更新节点状态和叙事
- 恢复过程对用户透明，不需要人类主动提供背景

## 5. 非功能需求

### NFR-1: Context 效率
- 全局视图 < 2k tokens
- 单个节点详细上下文 < 1k tokens
- 总系统开销（加载 + 管理）< 10k tokens

### NFR-2: 维护成本
- Agent 自主维护，不需要人类手动编辑结构化文件
- 人类通过自然语言交互即可创建、更新、查询节点
- 系统文件格式人类可读（Markdown / YAML）

### NFR-3: 渐进式
- 可以从 0 个节点开始，逐步添加
- 不需要预先定义完整的项目层级
- 结构随内容自然生长（分形：从原子任务长出项目，从项目长出战略）

### NFR-4: 容错
- 单个节点数据损坏不影响其他节点
- 缺少 parent 引用的节点自动归为顶层
- 状态不一致时以子节点为准，向上重新计算

---

## 6. 约束

- 存储：纯文件系统，Markdown + YAML，不依赖数据库
- 运行时：Agent 在 OpenClaw 框架内运行，通过文件读写和 memory 工具操作
- 兼容性：不破坏现有 MEMORY.md / memory/ 日志系统
- Token 预算：项目管理系统总占用不超过 context window 的 5%

## 7. 开放问题

1. **节点 ID 规则** — 人类可读的短 ID（如 `anext-001`）还是自动生成？
2. **归档策略** — done 后 7 天归档是否合适？还是按项目重要性区分？
3. **跨项目依赖** — 一个任务依赖另一个项目的任务，怎么表达？
4. **优先级** — 是否需要显式优先级字段？还是通过 deadline + status 隐含？
5. **度量** — 是否需要追踪完成率、耗时等量化指标？
6. **叙事体深度** — 时间线叙事写到什么粒度？每次交互都记还是只记关键节点？
