# Code Review Standard — CTO Agent 代码审查清单

*每次审查按此清单逐项过。不是建议，是流程。*

---

## 一、Spec 合规（必查）

- [ ] 代码是否符合 PRD 和 ARCHITECTURE.md？
- [ ] 状态机是否严格实现（只允许合法转换）？
- [ ] 数据模型是否跟 Schema 定义一致？
- [ ] 接口是否跟 INTERFACES.md 一致？
- [ ] 铁律测试是否覆盖所有 PRD 中的不变量？
- [ ] 有没有实现了规约外的功能（过度工程化）？

## 二、安全边界（必查）

- [ ] 金额计算有没有用 float？（必须整数/Decimal）
- [ ] 有没有绕过 Tool Call 直接写存储？
- [ ] 有没有硬编码密钥或敏感数据？
- [ ] 有没有直接操作生产环境的代码？
- [ ] **支付路径专项：**
  - [ ] 资金操作是否幂等？
  - [ ] 交易状态机是否不可跳跃？
  - [ ] 对账逻辑是否完整？
  - [ ] 并发场景是否安全？

## 三、代码质量

- [ ] 模块职责是否单一？有没有越界？
- [ ] 有没有循环依赖？
- [ ] import 风格是否统一（相对 vs 绝对）？
- [ ] 错误处理是否合理（不吞异常、不 bare except）？
- [ ] 注释是否符合标准：
  - [ ] 模块头部：做什么、不做什么、跟谁交互
  - [ ] 关键函数：为什么这样做
  - [ ] hack / 非直觉设计：当时的约束
- [ ] CLAUDE.md 是否跟代码同步？

## 四、架构健康

- [ ] 复杂度是否在 Complexity Budget 内？
  - 模块数是否超过上限？
  - 依赖边数是否合理？
  - 层级深度是否可控？
- [ ] 最大模块是否该拆？（经验值：>400 行考虑拆分）
- [ ] 新增模块是否有 Module Spec（含显式排除）？
- [ ] 架构图（OVERVIEW.md / ARCHITECTURE.md）是否已更新？

## 五、测试质量

- [ ] test:code 比是否合理？（目标 ≥ 1:1）
- [ ] 铁律测试是否完整且未被修改？
- [ ] 边界测试是否覆盖？（空值、极值、并发、异常路径）
- [ ] 测试是否可独立运行？（不依赖执行顺序）
- [ ] 有没有只测 happy path 漏掉 error path？

---

## 审查输出格式

```
## 审查结论（一句话）

## 🔴 严重问题（必须修）
- 编号、描述、修复建议

## 🟡 建议改进（可以后做）
- 编号、描述

## ✅ 做得好的地方

## 复杂度评估
- 模块数 / 最大模块行数 / 依赖深度 / 整体评价
```

---

## 审查频率

| 场景 | 审查方式 |
|------|---------|
| 新模块上线 | 完整审查（全部五项） |
| Bug 修复 | 针对性审查（二 + 五） |
| 重构 | 架构审查（四）+ 测试审查（五） |
| 定期健康检查 | 完整审查，每月一次 |
