# 1. 背景

## 1.1 AI Agent 的记忆困境

AI Agent 运行在一个根本性的矛盾之中：**它在单次对话内拥有超越人类的认知能力，但在跨越对话时面临系统性失忆。**

每次 Session 启动，Agent 面对的是一张白纸。没有昨天的记忆，没有上周的决策依据，没有对任何长线项目的全局熟悉感。它必须从文件中重建对世界的认知——而重建的速度、准确率和算力成本，完全受限于底层数据的组织范式。

这不是体验问题。这是 AI Agent 能否从「单次对话工具」进化为「具备长线执行力的协作者」的核心瓶颈。

> **类比：** 你每天早上醒来完全失忆，桌上放着一叠笔记。你能否安全、有效地继续昨天卡住的商务谈判，完全取决于这叠笔记的信息密度和因果逻辑。现在的问题是——传统纯文本笔记写得极差，业界的「智能检索库」又经常递错笔记，而当你试图整理笔记时，还可能因为「手抖」把关键信息撕毁。

## 1.2 当前架构的困局与业界盲区

面对长周期、多线程、强依赖的复杂项目时，现有的信息存储架构（包括业界主流方案）全面失效：

| 方案 | 用途 | 致命缺陷 |
|------|------|----------|
| `TODO.md` | 任务清单 | 扁平一维。无背景、无卡点、无依赖图谱。`- [ ] 审核信贷协议` 提供的信息增量接近于零 |
| `MEMORY.md` / 日志 | 长期记忆 | 按时间流水线而非业务实体组织。梳理单一项目历史需跨文件肉眼检索，决策链极易断裂 |
| 主流 RAG / 向量库 | 语义检索 | 懂语义，不懂业务因果律。能搜出「利率 7.8%」，但无法推断「法务审核卡住了商务谈判，必须物理阻断」，导致 Agent 产生逻辑幻觉 |
| 对话历史 | 临时工作记忆 | Token 极易耗尽（Lost in the middle），Session 结束即永久丢失，跨会话资产清零 |

**结论：** 纯文本清单承载不了网状业务逻辑；RAG 向量检索缺乏工程确定性。一旦遭遇多利益相关方、跨节点依赖的决策链，现有范式彻底失效。

## 1.3 核心问题：Context 管理与业务状态机失效

**一句话定义：** Agent 在处理复杂多线程项目时，无法在极低算力成本下获取绝对准确的业务拓扑，导致认知断裂、幻觉执行与财务失控。

拆解为 6 个子问题：

### 问题 1：认知断裂 — 不知道「发生了什么」

Agent 醒来面对任务「审核 Anext 信贷协议」，但不知道协议谈了多久、经历几个版本、还在等谁的回复。

**结果：** 要么反问用户（增加人类心智负担），要么凭有限上下文猜测（产生执行事故），要么从头处理（无效重复劳动）。

### 问题 2：因果链断裂 — 不知道「为什么」

即使 Agent 检索到当前状态（利率 7.8%），也不知道谈判底牌是什么、为什么担保条款存在风险。

**没有「为什么」，Agent 无法做出高质量判断。** 它极易给出与上周核心策略完全相悖的建议。

### 问题 3：全局视野缺失 — 不知道「还有什么」

Agent 专注于 Anext 谈判时，视野被极度收窄：看不到 Chris 的薪水方案明天就要 Deadline，也意识不到手头资源与其他项目的冲突。

**结果：** Agent 退化为被动应答器，丧失主动统筹、全局预警和发现跨项目卡点的 Agentic 特性。

### 问题 4：Context 污染与算力黑洞

为了弥补失忆，工程上的直觉是「全部塞给大模型」。但这带来双重灾难：

- **智商降级：** 无关噪音稀释 LLM 注意力（Lost in the middle 现象），推理质量断崖式下降
- **财务与性能暴走：** 每次对话全量加载 50k+ 历史日志，API Token 成本指数膨胀，首字响应延迟（TTFT）飙升至十数秒

### 问题 5：结构僵化 — 缺乏拓扑图谱与状态机

真实商业项目不是扁平清单，甚至不是树（Tree），而是**有向无环图（DAG）**：

- 任务之间有严格的跨项目阻断（A 未完成前，B 不能启动）
- 状态需要向上冒泡级联（底层子任务全 Done，父项目必须自动跃迁为 Done）

**用纯文本管理状态机，就像用记事本跑关系数据库的 JOIN 操作。** 底层数据结构的错配，直接锁死了系统复杂度的上限。

### 问题 6：写操作失控（Mutation Chaos）— 核心致死点

LLM 本质上是基于概率的文字接龙引擎。赋予它直接修改纯文本记忆文件的权限，它**一定会「手抖」**：

- 脏写：改了状态但忘了追加日志
- 幻觉：伪造不存在的任务 ID，甚至造出 A→B→C→A 的循环死锁

**结果：系统不仅会失忆，还会产生「虚假记忆」。** 人类不仅不敢信任 Agent 的决策，甚至不敢信任 Agent 写下的底层数据。

## 1.4 问题的影响面

| 影响维度 | 真实表现 | 后果 |
|----------|----------|------|
| **效能倒挂** | Jeff 每次交互前都需要向 Agent 补充前情提要 | 本应由 AI 节省的时间反而成倍增加 |
| **执行灾难** | Agent 缺乏全局因果链与物理阻断常识 | 可能在法务 Block 的情况下强行推进，引发商业事故 |
| **遗漏风险** | 系统缺乏独立于对话之外的后台提醒 | Deadline 被无声错过，异步任务陷入永久停滞 |
| **算力黑洞** | 无差别加载海量无关文本 | API 成本指数膨胀，系统响应被拖垮 |
| **信任破产** | Agent 频繁忘事、写坏底层记忆、逻辑自相矛盾 | 用户彻底退回手动管理 |

**终极定论：** 如果不能将「确定性的状态流转」与「概率性的 LLM 思考」在架构上严格解耦，Agent 就永远只是一个高昂的聊天工具，无法成为具备生产交付能力的核心中枢。

## 1.5 核心用户与交互不对称性

本系统存在严重的人机交互不对称性，设计必须承认并利用这一点：

- **主用户：AI Agent（执行侧）**
  - 系统每天高频交互的执行者，它的用户体验即系统的接口体验
  - 核心诉求：零噪音上下文注入（DCP 推流）、原子化变更工具（Tool Calls 防呆拦截）、底层自动计算状态冒泡和死锁检测的护航引擎

- **次用户：Jeff（决策侧）**
  - 核心诉求：
    1. **自然语言交互：** 用人话下指令、查询、做决策，不需要学习任何命令或语法
    2. **全局可见性：** 随时掌握所有项目的进展、风险、卡点，无需自己翻文件拼凑
    3. **数据主权：** 不需要手动维护底层结构图谱，但所有数据保持纯文本可读、Git 可追踪，随时可进行人工审计或版本回滚

# 2. 设计哲学：向人类认知学习

## 2.1 人类 Context 管理机制及启示

人类能在有限工作记忆下处理复杂任务，不是因为记得多，而是因为具备高效的认知压缩、注意力分配和外部记忆协作能力。本系统借鉴其中最有效的机制，将其转化为 Agent 可执行的系统能力。

| 人类机制 | 原理 | 可借鉴度 | 系统设计启示（FPMS 系统级补偿） |
|----------|------|----------|-------------------------------|
| **分块（Chunking）** | 工作记忆 4-7 块，每块可压缩复杂概念 | ✅ 完全映射 | 每个节点是一个认知块，一句话摘要即可恢复对该节点的认知 |
| **叙事（Narrative）** | 故事比清单好记 10 倍，天然携带因果链 | ✅ 完全映射 | Context 以时间线叙事体组织，保留因果链和决策背景，绝不用无序 bullet |
| **遗忘（Forgetting）** | 主动过滤噪音，是功能不是 bug | ✅ 机制模拟 | 已 Done 且无入度依赖的节点自动降权冷冻，保持工作区 Index 纯净 |
| **周期回顾（Rehearsal）** | 周会/复盘对抗遗忘曲线 | ✅ 机制模拟 | Heartbeat = 系统替 Agent 开的「数字周会」，主动揪出风险项 |
| **外化（Externalization）** | 承认记不住，借助笔记/日历/看板 | ✅ 完全映射 | 本地文件系统（SQLite+MD）+ 全局概览 = Agent 的外化「数字白板」 |
| **关联触发（Association）** | 线索触发记忆链，一环拉出一长串 | ⚠️ 系统代偿 | Agent 无直觉，靠底层 DAG 图计算引擎实现确定性关系推导补偿 |
| **聚焦（Focus）** | 任何时刻只深度关注一件事 | ✅ 机制模拟 | 焦点节点加载 100% 全量上下文，其余物理降级为一句话摘要 |
| **联想打包（Bundling）** | focus 时大脑自动拉入相关背景 | ⚠️ 控制反转 | 摒弃 Agent 主动检索。由脊髓引擎将 parent/依赖方拼装打包，直接喂给 Agent |
| **视觉分辨率（Foveal）** | 注视点高清，周边模糊但画面完整 | ⚠️ 机制模拟 | 焦点=中央凹（全量），近景=周边视觉（摘要），全局看板=整体大盘永不丢失 |

## 2.2 大模型与人类大脑的关键差异

设计不应只学习人类认知的优点，也必须正视 Agent 与人类在记忆机制上的根本差异。

| 维度 | 人类（碳基大脑） | 我们的智能体系统（LLM + 底层基建） |
|------|-----------------|----------------------------------|
| Session 内记忆 | 极有限（4-7 chunks） | 极强（200k+ window），且通过多分辨率受控 |
| 跨 Session 记忆 | 持续存在，模糊但不消失 | 通过本地文件与数据库图谱实现物理持久化 |
| 检索方式 | 关联触发，潜意识自动浮现 | 摒弃工具摸黑搜索，采用确定性主动推流（DCP）极速注入 |
| 遗忘 | 自动、智能、情绪加权 | 基于图谱入度规则与时间衰减的强制冷冻 |
| 压缩 | 无意识自动完成 | 需要显式设计（如小模型异步跑批处理压缩） |
| 背景意识 | 模糊但持续在线 | 通过三层分辨率（焦点/近景/背景看板）强行构建 |
| 聚焦 | 自动、直觉驱动 | 需要底层状态机（Deadline/卡点）进行中断与强制调度 |

### 核心洞察

1. **人类有持续的背景意识，LLM 要么全知要么全盲。** 系统必须用「多分辨率眼球模型」弥补这个根本差异。
2. **人类天生知道"现在最该关注什么"，LLM 没有这个直觉。** 必须通过底层心跳或显式指令强制分配注意力。
3. **人类的注意力像眼球——注视点高清，周边模糊，但整体画面永不丢。** 系统需要「全量叙事 / 状态摘要 / 一句话标题」的连续分辨率梯度，而非「加载或不加载」的二元开关。
4. **人类 focus 时大脑自动联想打包，LLM 不会。** 必须由底层代码（脊髓）代为完成图谱遍历，构建好「认知包（Context Bundle）」直接推给 LLM，绝不能让 Agent 浪费算力去自己 Pull。

## 2.3 硅基 vs 碳基：系统的独特优势

> 注意：这里的优势不单指 Agent（LLM），而是指「LLM 大脑 + 结构化底层系统」的完整混合体。

| 维度 | 人类（碳基） | 智能体系统（硅基混合体） |
|------|-------------|------------------------|
| 回忆精度 | 模糊、会变形、会自我编造 | 100% 精确，写下即可万次无损读取 |
| 并行逻辑 | 极弱（心算超过 3 步必错） | LLM 极弱，但底层代码状态机可 100% 严谨地瞬间完成几十个节点的级联冒泡 |
| 结构化处理 | 费劲，极度消耗精力 | 机器天然擅长遍历 DAG，对 YAML/SQL 读写如飞 |
| 情绪偏差 | 焦虑与疲惫会扭曲判断 | 纯粹基于权重和 Deadline 数学规则执行 |

### 终极推论：为什么少数精准 Chunk 胜过海量原始数据

人类只有 4-7 个 chunk 就能完成极其复杂的任务。不是因为记得多，而是因为**每一块都是精准的、高度抽象的**。

外科医生做手术时脑子里不是整本解剖学教科书，而是精确到这一刀需要的那几个关键信息。Agent 应该追求同样的精准度。

**这个类比可以精确展开：**

| 维度 | 外科医生（人类认知） | FPMS 系统（Agent 认知） |
|------|---------------------|------------------------|
| 手术前 | 阅读完整病历、影像、化验报告 | 底层脊髓引擎遍历完整图谱、依赖链、历史日志 |
| 手术中脑子里有什么 | 只有这一刀的关键坐标：切入点、避让神经、止血位 | 只有当前任务的认知包：焦点节点全量 + 依赖摘要 + 风险标记 |
| 手术中脑子里没有什么 | 不会回想昨天的晚餐、不会翻阅教科书第3章 | 不会加载已归档节点、不会塞入无关项目的历史 |
| 关键信息从哪来 | 护士递器械、麻醉师报数据——**被投喂，不是自己找** | 脊髓引擎打包推送——**DCP，不是 RAG Pull** |
| 全局画面 | 余光看监护仪（心率、血压、血氧）——模糊但持续在线 | 背景看板（全局概览）——一句话状态，永不丢失 |

**为什么这种"精准提纯"在百万 Token 时代反而更重要：**

1. **注意力是零和博弈。** Transformer 的 Softmax 注意力权重总和恒等于 1。塞入的 token 越多，每个关键 token 分到的注意力权重越低。这是数学，不会因为窗口变大而消失。无关信息不是中性背景板——它是物理层面的认知干扰弹。

2. **"找得到"不等于"想得清"。** 模型能在 1M token 里找到一根针（检索），不代表它能在 1M token 的噪声里完成多步因果推理。检索和推理是完全不同的认知任务，后者对信噪比的要求高几个数量级。

3. **延迟是物理墙。** Agent 的核心是高频 OODA 循环（感知→思考→行动→观察）。每次循环的 TTFT（首字延迟）与输入长度正相关。塞满 1M 意味着每次思考前要等几十秒——系统瘫痪。

4. **成本是线性的。** 输入 token 成本与长度线性相关。Agent 每天执行上百次循环，无差别塞满 vs 精准提纯，年成本可能差 100 倍。

5. **1M 是仓库容量，不是工作台面积。** 长上下文的真正价值在于"可访问全集"（低频精读一本厚书），不在于"每次循环都把全集塞进大脑正中央"。真正送进推理核心的，仍然是小而精的认知包。

**因此，系统的使命是：**

> 在当前任务下，以最小必要上下文，提供最高信噪比、可推理、可执行的认知包。Memory 系统的本质不是存储——存储已经不稀缺了——而是**注意力分配**。

## 2.4 设计原则

1. **控制反转（IoC）** — 默认由系统完成依赖梳理、状态归集与上下文打包，Agent 优先消费已整理结果，而不是每轮从原始材料中自行检索重建。必要时允许 Agent 请求扩展检索，但不承担底层编排责任。
2. **最小充分上下文（Minimum Sufficient Context）** — 每次只注入完成当前任务所必需的信息集合。目标不是最小化 token 数，而是在不损失判断质量的前提下，最小化无效上下文。1M context 是仓库容量，不是工作台面积——系统应根据任务复杂度动态调整注入量，而非硬性截断。
3. **精准优于全面** — 高信噪比、任务相关、组织良好的 Context，优于大体量但松散冗余的 Context。窗口增大的红利应用于保留更关键的证据与因果，而不是堆积更多噪音。
4. **抽象是系统的生命线** — 系统必须持续将旧日志、历史过程和低频细节压缩为多层摘要，使长期记忆可持续、可恢复、可推理。节点摘要质量决定系统效率上限。
5. **分形自相似（Fractal）** — 从原子任务到战略目标，系统采用统一的核心抽象与关系模型，并允许在不同层级做有限扩展，以降低解析复杂度并支持递归增长。
6. **算力与模式解耦（Brain-Spine Model）** — 任何可由规则稳定计算的问题，都不应把正确性依赖在概率模型上。状态流转、依赖解析、死锁拦截、投影生成等确定性逻辑由底层运行时完成；大模型仅负责理解业务语义、提出策略与生成表达，并通过受约束的工具接口与底层系统交互。
7. **动作空间收敛（Constrained Mutation）** — Agent 是概率系统，时间拉长必然产生长尾错误。因此必须绝对剥夺 Agent 对底层数据的自由覆写权——所有变更必须收敛为极少数带强类型校验的原子化 Tool Calls。任何违背业务因果律的动作（如制造循环依赖、误删关键卡点），由底层脊髓引擎硬拦截。

# 3. 目标

## 3.1 核心目标

本系统要解决的不是"项目管理软件"的问题，而是"AI Agent 如何在跨 Session 场景下持续、高效、可控地管理复杂项目群"的问题。

### G1. 跨 Session 认知连续性

Agent 每次唤醒时，能在**秒级**（而非分钟级）恢复对所有项目的全局认知——包括当前状态、关键卡点、待办优先级和近期决策脉络。消除"每次对话从零开始"的根本瓶颈。

### G2. 最小充分上下文注入

系统根据当前任务自动组装**最高信噪比**的认知包（Context Bundle），精准推送给 Agent。不多不少——既不因信息缺失导致错误决策，也不因冗余加载浪费注意力和算力。

### G3. 确定性与概率性解耦

所有可由规则稳定计算的工作（状态流转、依赖冒泡、死锁检测、投影生成）由底层脊髓引擎确定性完成，**系统正确性不依赖 LLM**。大模型仅负责业务理解、策略判断和自然语言输出。

### G4. 全局视图始终准确

任何节点的状态变更，自动沿依赖图谱向上冒泡，保持全局看板、项目摘要和父节点状态**近实时一致且可重建**。无需人工维护，无需 Agent 逐层手动更新。

### G5. 分形可扩展

从原子任务到跨业务线的战略目标群，系统使用统一核心 Schema 与同一套引擎。结构应随业务增长保持可控扩展，避免因节点层级和依赖关系增长而导致复杂度失控。

### G6. 次用户（人类监督者）体验

人类监督者无需学习任何命令或语法：
- **自然语言交互**：说人话即可下达指令和查询
- **全局可见性**：随时掌握所有项目的进展、风险、卡点
- **数据主权**：系统数据可审计、可导出、可回滚；叙事内容以 Markdown 保存，结构化状态以 SQLite 管理，关键变更可通过 Git 或事件日志追踪

### G7. 成本可控（FinOps）

系统级 token 管理开销（Context Bundle 组装 + 全局投影）占单次 Agent 调用总 token 的比例可量化、可优化。避免因项目管理上下文无节制膨胀导致 API 成本失控。

### G8. 可恢复性

所有全局视图、焦点列表与认知包都必须可由事实源重建。系统在任意时刻都可恢复到一致状态——派生视图损坏不影响数据完整性，重建即可修复。其中，结构化状态与关系以 SQLite 为事实源，叙事历史以 Markdown 为事实源；所有摘要、投影与认知包均为派生物，不构成第二真相源。

## 3.2 非目标（v1 不做）

| 非目标 | 原因 |
|--------|------|
| 可视化仪表盘（Dashboard） | v1 聚焦 Agent 内部认知引擎；人类监督通过自然语言 Tool Call 实现，无需独立 UI 层 |

# 4. 功能需求

## FR-0: 事实源与派生物边界

系统中所有数据分为**事实源**与**派生物**两类，工程实现时必须严格区分，禁止边界漂移。

### 事实源（Source of Truth）

事实源分为三层：

**1. 业务事实源**
- 结构化节点骨架（SQLite `nodes` 表）：节点标识、主状态、业务元数据
- 关系图谱（SQLite `edges` 表）：显式区分强边（parent，垂直归属）与弱边（depends_on，横向依赖）
- 叙事历史（Markdown append-only 文件）：节点的时间线与因果记录

**2. 运行时事实状态（持久化）**
- 焦点列表（SQLite `session_state` 表）：当前 Agent/人类显式指定的注意力指针。焦点包含显式主观意图与自动调度结果，不能仅由图谱拓扑唯一推导，必须持久化以保证跨 Session 不失焦。

**3. 审计事实源（v1 必须实现）**
- 事件日志（events.jsonl）：事务级写入痕迹，用于追踪、补写与灾难恢复
- repair 记录：MD 写入失败时的修复队列

### 派生物（Derived / Rebuildable）

- 全局看板视图
- 认知包（Context Bundle）
- 风险标记（查询时动态计算，不持久化）
- 归档索引
- 各类缓存与投影视图

派生物可在任意时刻由事实源重建。派生物损坏、丢失或过期不构成数据丢失，重建即可修复。**派生物不作为人工编辑对象**——系统允许其失效和重建，但不允许以手工修改派生物的方式修正业务事实。

### SQLite 表分类

工程实现时须明确区分：

- **事实表**：`nodes`、`edges`、`session_state`
- **派生表 / 缓存表**：`narrative_index`、`global_view_cache`、`risk_cache`、`archive_index`

若某表内容可由事实源全量重建，则不得归类为事实表。

### 系统不变量（Invariants）

#### 数据一致性不变量（由脊髓引擎在事务内强制校验）

1. 任一节点最多只有一个 `parent_id`。（违反时：回滚事务，拒绝写入）
2. `depends_on` 图不得成环（DAG 约束）；新增或修改依赖关系时必须执行环路检测。（违反时：拦截写入，抛出错误）
3. 所有事实层变更必须通过受约束 Tool Call 或系统确定性事务完成，禁止绕过写入协议直接修改底层存储。（违反时：拒绝操作）
4. 风险标记由脊髓引擎动态计算，不得由 LLM 或人工直接写入。（违反时：忽略输入，重新计算）
5. 任何骨架事实变更必须在提交时同时生成审计记录（events.jsonl）；叙事层（MD）可同步追加，若失败则进入 repair 流程补齐。禁止"改状态无审计轨迹"。（违反时：回滚事务，拒绝写入）
6. SQLite 事务提交为系统主提交点；MD 写入失败记录 repair event，不回滚已提交的结构化状态。（MD 修复由后续 repair 流程完成）

#### 系统运行不变量（由架构和运行时机制保证）

7. 派生物不得作为事实修改的输入依据；事实源变更只能由受约束 Tool Call 或系统事务性维护逻辑直接作用于事实层。
8. 任一派生物删除、损坏或过期，不得导致业务事实丢失；系统必须提供从事实源重建该派生物的确定性路径。（实现要求：检测到派生物异常时，自动触发重建并记录日志）
9. 派生物不作为人工编辑对象，不允许以手工修改派生物的方式修正业务事实。系统可在下一次重建时无条件覆盖手工修改。

## FR-1: 统一分形节点模型

所有层级（战略目标 / 项目 / 里程碑 / 任务）使用统一核心 Schema，支持无限嵌套的自相似结构，并允许节点类型在展示与业务规则上有轻微差异。

**节点类型枚举**：`goal` / `project` / `milestone` / `task` / `unknown`（Inbox 节点创建时默认为 `unknown`，挂载时应明确为具体类型）

类型用于语义标注和显示优化，不影响底层图谱引擎的统一处理逻辑。

**节点字段分级**：

| 级别 | 字段 | 所有权 | 说明 |
|------|------|--------|------|
| 系统标识 | id, created_at | 系统自动生成 | 节点身份，不可修改。id 格式：语义前缀 + 短哈希（如 `task-7f2a`），兼顾防冲突、人类可读与 LLM token 效率 |
| 生命周期 | status | Agent 通过 Tool Call 变更，系统校验合法性 | 受约束业务状态 |
| 创建时指定 | title, node_type | Agent/人类在创建时提供，后续可通过 Tool Call 受约束修改 | title 用于命名与检索 |
| 关系 | parent_id（强边）, depends_on（弱边） | 通过 Tool Call 管理，系统校验拓扑约束 | 存储在 `edges` 关系表中 |
| 认知核心 | summary | Agent/人类/压缩器可写（人工摘要优先级高于压缩器自动生成，压缩器不得覆盖人工编辑的摘要） | **L0 认知接口**：节点在全局看板和快速恢复中的最小表示 |
| 业务 | why, next_step, owner, deadline | Agent/人类可写 | 决策支持与执行指引 |
| 系统维护 | updated_at, status_changed_at, archived_at | 系统在事务中自动维护 | 冒泡、归档、衰减计算 |

**字段语义分工**：
- `title` 用于命名与检索（"这个节点是什么"）
- `summary` 用于快速恢复当前认知（"这个节点现在最重要的状态是什么"），不应简单重复标题
- `owner` 指当前主要推进责任主体，而非所有相关方列表

**节点生命周期最小条件**：

| 阶段 | 最小条件 |
|------|----------|
| 创建（inbox） | id, title, status=inbox, created_at |
| 进入活跃图谱 | 必须具备 summary；status ≠ inbox；必须具备 parent_id 或被显式声明为顶层根节点（root） |

**核心约束**：
- 每个节点最多一个 parent_id（树形归属）
- 节点可存在多个 depends_on 依赖关系，允许跨项目横向依赖；工程实现以 `edges` 关系表建模
- 节点允许在创建初期处于未挂载状态（无 parent_id），但进入活跃图谱前应满足最小归属条件
- 每个节点是一个认知"块"——summary 足以快速判断节点当前意义与是否需要展开详情

## FR-2: 叙事体上下文（Append-Only Narrative）

节点的历史记录采用必须 append-only 的时间线叙事体，作为业务因果链的可读历史轨迹，而非无序列表。

### 物理组织

每个节点对应一个独立叙事文件（如 `narratives/{node_id}.md`）。叙事文件与节点通过 node_id 一一绑定，脊髓引擎按 node_id 确定性定位和提取。

### 叙事条目结构

每条叙事条目包含以下最小元信息，由脊髓引擎确定性组装后落盘：

| 字段 | 来源 | 说明 |
|------|------|------|
| timestamp | **系统注入**（OS 时钟，带时区偏移 ISO 8601） | 格式如 `2026-03-13 16:00 +08:00`。兼顾机器精确解析与人类本地阅读。Tool Call 签名中禁止包含日期参数 |
| actor | Agent 传入 | 事件主体/来源（如 Jeff、Charles、律师、系统自动） |
| event_type | 由 Tool 类型决定或 Agent 传入，系统校验枚举合法性 | `update_status` → 固定 `status_change`；`append_log` → 允许 `info` / `decision` / `blocker` / `correction` |
| fact | Agent 传入 | 触发事实 |
| reason | Agent 传入（状态变更时必传） | 该事件的业务意义与直接后果，不是简单重复事实 |

**落盘格式**（由脊髓引擎用代码模板确定性拼装，Agent 无权自行拼接格式字符串）：

`- [YYYY-MM-DD HH:MM ±ZZ:ZZ] [actor] fact → reason (event_type: state_delta)`

状态变更时，`state_delta`（如 `active→waiting`）由脊髓引擎自动对比注入，无需 Agent 传入。纯信息事件省略 state_delta 和空字段。

示例：
```
- [2026-03-13 14:20 +08:00] [Charles] 回复可接受7.8%但坚持担保条款 → 价格让步但风险未解除 (info)
- [2026-03-13 16:00 +08:00] [Jeff] 决定发律师审核担保条款 → 等待律师意见再推进 (status_change: active→waiting)
- [2026-03-15 10:30 +08:00] [系统] [更正: 2026-03-13 14:20] 3/13记录有误，实际报价为7.5%非7.8% (correction)
```

### 写入规则

**写入路径分流**：
- **状态跃迁**（强因果绑定）：调用 `update_status` 时必须传入 `reason`，脊髓引擎在同一事务内完成骨架变更 + 叙事追加。event_type = `status_change`
- **纯事实叠加**（有边界的宽容）：通过 `append_log` 记录未改变主状态但会影响判断、风险、依赖或下一步动作的关键业务事件（如"法务退回要求修改，状态仍是waiting"）。不用于转储日常闲聊、重复确认或低价值噪音。高频事件的降维由异步压缩器（FR-12）负责

**不可变性约束**：
- 必须 append-only，不允许覆盖或删除历史内容（违反时：拒绝操作）
- 修正错误记录必须追加 `correction` 类型条目，保留纠偏轨迹，不删除原始条目
- 所有写入通过受约束 Tool Call 完成，Agent 不得直接编辑叙事文件，也不得自行生成未结构化文本写入

**膨胀控制**：
- 单节点叙事条目超过 50 条或预估 token 超过 2000 时，系统标记该节点为"建议压缩"（阈值为可配置默认值）
- 压缩由 FR-12 异步执行：原始叙事不删除，仅降低在认知包中的可见优先级

### 叙事与骨架的关系

- 叙事负责"可理解"（供 LLM 和人类阅读），骨架负责"可计算"（供脊髓引擎查询推导）
- 两者互补，叙事不替代骨架字段作为状态事实源
- **冲突时以骨架为准**：当叙事内容与结构化骨架字段不一致时，以骨架字段为当前状态事实，叙事作为历史解释层
- 叙事体是面向认知恢复的业务事件层，不是 Tool Call 审计日志（events.jsonl）的原样转储

## FR-3: 全局看板投影

脊髓引擎维护一个紧凑的全局视图，作为 Agent 的背景余光（L0 Peripheral Vision）。属于派生物，可从事实源随时重建。

**生成方式**：由脊髓引擎从 SQLite 确定性生成，非人工维护，非 LLM 生成。

**刷新触发**：
- 任何 Tool Call 导致节点状态变更后（实时重绘受影响区域）
- Session 启动时（全量重建）
- Heartbeat 周期（异步全量刷新）

**覆盖范围**：所有未归档且对当前管理仍有意义的节点，包括 inbox、active、waiting 以及带风险标记的相关节点；排除 dropped 与已归档节点。

### 结构化渲染（保留分形拓扑）

**树形渲染铁律**：看板必须基于 parent_id 强边关系渲染为带层级缩进的树形结构，拒绝扁平流水账。排序规则仅在同一层级的兄弟节点之间生效。

**布局分区**：
- **Zone 0（置顶区）**：所有 status=inbox 且 parent_id=null 的节点，脱离业务树悬浮在看板顶部，形成视觉刺点
- **Zone 1（活跃业务森林）**：已挂载且未归档的业务节点，按树形缩进渲染

**每行展示格式**：`[status_icon] {node_id}: {title} (风险标记 / deadline)`。看板中不展开 summary 或叙事正文，保持最高信息密度。

渲染示例：
```
[收件箱] 明天打电话给会计
[收件箱] 查一下NetStar合同细节

📁 Anext信贷 [active] 🚨at-risk
  ├─ 法务审核 [waiting] blocked
  ├─ 担保条款谈判 [active] 🚨deadline:3/18
  └─ 内部审批流程 [done]
📁 NetStar合作 [active]
  └─ 技术对接 [waiting] stale
```

### Token 预算与截断

**预算**：目标 2-3k tokens。

**截断策略**（当活跃节点过多触及预算上限时）：
1. 优先折叠无风险标记的健康分支——将深层叶子节点替换为聚合行（如 `... [折叠] 8 个正常推进子项`）
2. 高危节点（blocked/at-risk/stale）及其祖先路径强制展开，不可被折叠
3. 最低优先级：完全折叠为项目级计数

### 排序规则

v1 采用两阶段规则排序：

**阶段一：顶级项目排序**（根据子树内最高风险烈度降序）：

| 优先级 | 条件 | 说明 |
|--------|------|------|
| 1 | 子树内存在 blocked 且被活跃节点依赖 | 最高危战区 |
| 2 | 子树内存在 at-risk | 临期战区 |
| 3 | 子树内存在 stale | 停滞战区 |
| 4 | 无风险标记 | 按子树最近 status_changed_at 排序 |

**阶段二：同级兄弟排序**（同一父节点内，按以下优先级降序）：

| 优先级 | 条件 |
|--------|------|
| 1 | blocked 且被活跃节点依赖 |
| 2 | at-risk |
| 3 | stale |
| 4 | 其余按 status_changed_at 降序 |

## FR-4: 确定性上下文推送（DCP）

必须采用 Push 模式，严格遵循控制反转（IoC）原则。Agent 不得绕过脊髓引擎直接对事实源进行自由检索。Agent 可通过受约束接口请求上下文扩展或焦点切换。

DCP 的直接输入为当前焦点集合及其优先级。脊髓引擎基于焦点自动组装认知包（Context Bundle）并推送给 Agent。

> 本条定义 DCP 的原则、分辨率模型与装载协议；具体组装规则见 FR-10。

**推送触发时机**：
- Session 启动时（Bootstrap Context）
- 焦点切换时（Saccade 重建）
- Agent 请求扩展时
- Heartbeat 扫描发现新告警时

**上下文三级分辨率**（对应眼球模型）：

| 层级 | 信息密度 | 内容 | 典型 token |
|------|----------|------|------------|
| L0 背景 | 一句话状态 | 全局看板（FR-3） | ~500-1k |
| L1 近景 | 摘要级 | 焦点自身摘要、parent 目标、未完成子节点状态、siblings 状态、depends_on 前置方状态与卡点、**depended_by 后置方状态**（谁在等我完成——爆炸半径评估） | ~1-3k |
| L2 焦点 | 工作级上下文 | 焦点节点骨架全字段 + 摘要层 + 最近关键叙事（默认最近 3 天或最近 5 条）；完整原始叙事仅在受约束扩展时加载 | ~2-5k |

**受约束扩展接口**：
- `expand_context(node_id, mode)` — mode 指定扩展维度：
  - `history`：加载更多历史叙事（向前回溯）
  - `relations`：展开更多关联节点到 L1/L2
  - `full`：加载焦点节点的完整原始叙事
- 扩展有上限（单次扩展不超过 5k tokens），防止退化为无限制检索

**约束**：认知包总量遵循最小充分上下文（MSC）原则。经验带宽参考：概览 ~1-5k，执行 ~5-20k，复杂依赖 ~15-20k，战略回顾 ~30-50k。

**降级策略**：当认知包超预算或局部上下文不可用时，按 L2 → L1 → L0 顺序受控降级——优先保留 L0 和焦点节点最小可工作集（骨架 + summary），不失败退出。焦点无效时 fallback 到纯 L0 全局模式，等待用户指令。

## FR-5: 状态引擎与级联推导

### 5.1 节点状态

状态集：`inbox` / `active` / `waiting` / `done` / `dropped`（排他性 ENUM，默认 inbox）

主状态通过受约束 Tool Call 显式变更；系统负责校验状态迁移合法性，但不通过推导逻辑直接覆盖节点主状态。

**合法状态迁移**：

```
inbox → active, waiting, dropped
active → waiting, done, dropped
waiting → active, done, dropped
done → active（重开，需 reason_log）
dropped → inbox（恢复，需 reason_log）
```

迁移前置条件（由脊髓校验，违反时拒绝）：
- inbox → active：必须已有 summary 且具备 parent_id 或 root 声明
- → done：建议具备 summary（记录完成意义）
- done → active / dropped → inbox：必须传入 reason_log（解释重开/恢复原因）

### 5.2 风险标记（动态计算属性）

风险标记不持久化存储，由脊髓引擎在查询或组装认知包时基于事实层状态与当前时钟（UTC）实时计算：

| 标记 | 计算规则 | 说明 |
|------|----------|------|
| `blocked` | 存在至少一个 depends_on 前置节点，其主状态 ≠ done | 横向阻塞。**dropped 不解锁**——前置依赖被废弃意味着死胡同，不是通行证。判断基于事实层 status，不受归档影响 |
| `at-risk` | deadline < NOW() + 48h 且 status ∉ {done, dropped} | 时间风险（v1 仅实现时间维度） |
| `stale` | status ∈ {active, waiting} 且 status_changed_at < NOW() - 7 days | 僵尸风险——长期无有效推进 |

风险标记可与任何主状态叠加。LLM 无权直接设置风险标记。风险标记是纯计算属性——不存在"设置"或"解除"操作，只有底层条件变化后重新计算的结果。

### 5.3 垂直冒泡（强边 parent_id）— rollup_status

系统为父节点自动推导 `rollup_status`（派生属性，区别于父节点自身的 `status`）。

**按优先级求值**（从上到下匹配第一个命中的规则）：

1. 无子节点 → rollup_status = 自身 status
2. 存在 active 子节点 → rollup_status = active
3. 存在 waiting 或 inbox 子节点 → rollup_status = waiting
4. 所有子节点为终态（done 或 dropped 混合）：若存在至少 1 个 done → rollup_status = done
5. 所有子节点为 dropped（100% 全线撤退）→ rollup_status = dropped

**风险暴露**：任一子节点存在风险标记时，父节点生成 `has_risk_children = true` 及风险来源摘要（哪个子节点、什么风险），供全局看板与认知包展示。不将子节点风险标记直接复制为父节点同名标记。

`rollup_status` 供全局视图和认知包展示使用；父节点自身的 `status` 由人工/Agent 管理，不被系统自动覆盖。

**双轨呈现**：认知包中必须同时展示父节点的 `status`（主观意图）与 `rollup_status`（客观推导）。当两者背离时（如 status=active 但 rollup_status=done），这种张力将自然引导 Agent 执行对齐动作（如写结项总结并关单）。

### 5.4 横向级联（弱边 depends_on）

**依赖状态变化时**：当图谱中任意节点状态变更后，脊髓引擎在后续查询中重新计算所有受影响节点的风险标记。若某节点的全部前置依赖均已完成（done），则 `blocked` 风险在下次计算时自然消失。若前置依赖被 dropped，blocked 持续存在且升级为"死胡同告警"——Agent 应决定是否移除该依赖或调整计划。

**唤醒提示**：当前置卡点状态发生变化（完成或废弃）时，脊髓引擎主动生成虚拟中断事件注入受影响节点的下一次认知包（如 `🚨系统提示：前置卡点 {node_id} 已扫清，请评估推进` 或 `🚨系统提示：前置依赖 {node_id} 已废弃，请评估是否调整计划`），确保 Agent 不会遗漏依赖图的变化。

**新建 depends_on 时**：
- 脊髓引擎执行 DAG 环路检测，发现成环立即拦截，拒绝写入
- 若目标节点状态 ≠ done，当前节点在下次风险计算时将自然呈现 `blocked`

**状态变更副作用清单**（节点变为 done 时的完整处理链）：
1. 脊髓在事务内更新骨架状态 + 追加叙事
2. 后续查询中：所有 depended_by 节点重新计算 blocked 状态
3. 父节点 rollup_status 重新推导
4. 全局看板触发刷新
5. 若满足 FR-6 归档前置条件，进入归档候选

### 5.5 执行约束

- **写路径**：所有关系合法性校验（环检测、状态迁移合法性）与事实层变更由脊髓引擎在事务内确定性完成
- **读路径**：所有派生状态（rollup_status）与风险标记（blocked/at-risk/stale）由脊髓引擎基于事实层确定性推导
- LLM 仅下达状态变更指令，不参与任何推导计算

## FR-6: 拓扑安全归档

作为注意力层级迁移机制，已结束的节点在拓扑安全条件满足后从活跃认知空间退出，释放全局视图与认知包的 token 空间。

**归档扫描触发**：脊髓引擎在 Heartbeat 周期中执行归档候选扫描。

**归档触发条件（必须同时满足）**：

| 条件 | 规则 | 说明 |
|------|------|------|
| 1 | status ∈ {done, dropped} | 生命周期已结束 |
| 2 | status_changed_at < NOW() - 7 days | 冷却期（可配置默认值）。v1 对 done 与 dropped 采用统一冷却期，后续版本可按终态类型细化 |
| 3 | 不存在来自任何**未归档**节点的 depends_on 依赖（无论其状态） | 只要热区有任何节点依赖它，就必须留在热区陪跑，杜绝悬挂指针 |
| 4 | 不存在任何**未归档**的子孙节点（无论其状态） | 必须从叶子开始归档，自底向上。防止父节点先归档导致子节点 parent_id 悬空 |

**归档效果**：
- 事实层：设置 `archived_at = NOW()`，标记为归档状态
- 派生层：活跃索引在事务后派生视图刷新时移除（即使刷新延迟，归档事实仍成立，后续可重建）
- 归档节点默认不参与全局投影与 Bundle 组装，仅在显式搜索、手动恢复或历史追溯场景中被加载
- 叙事文件和骨架记录完整保留

**归档恢复**：通过 `unarchive(node_id)` Tool 逆操作。归档节点主状态重新变为非终态时，自动返回活跃索引并重新参与全局投影与 Bundle 组装。脊髓校验拓扑合法性后执行恢复。

**常驻豁免**：关键战略节点可通过 `is_persistent = true`（由 Tool 设置）标记为常驻，豁免自动归档。常驻仅影响归档资格，不影响终态判定。

**例外**：关键战略节点可由人类监督者标记为"常驻"，豁免自动归档。

## FR-7: 统一收件箱（Inbox）

Inbox 是正式图谱之前的受限缓冲层。对话中产生的模糊待办，Agent 通过 Tool Call 创建标准节点，status = `inbox`，parent_id = null。

**不使用独立的 TODO.md 文件**——Inbox 是分形节点模型的一部分，不引入第二种数据格式，杜绝双重事实源。

**创建约束**：
- 最小必填字段：`title`（其余可选，降低捕获摩擦）
- node_type 默认为 `unknown`，整理时需明确

**Inbox 节点特征**：
- 悬浮在全局看板 Zone 0 顶部，保持可见。当 Inbox 数量超过看板预算时，仅展示最近创建与最久未整理的前 5 条，其余以计数摘要呈现（如 `+12 inbox 未整理`）
- Inbox 节点在整理完成前，**不参与任何 depends_on 依赖关系**（既不允许正式节点依赖 inbox，也不允许 inbox 依赖正式节点），**不参与父节点 rollup 计算**——未分拣对象不承担正式图谱职责。脊髓引擎对跨越隔离区的 depends_on 写入执行硬拦截
- 整理由 Agent 在后续交互中通过受约束 Tool Call 完成（`attach_node` 设置 parent_id + 变更 status）；系统仅负责暴露和提醒，不自动为 Inbox 推断 parent 或正式状态

**整理完成条件**（满足任一）：
- 已设置合法 parent_id 且 status ≠ inbox
- 或被显式声明为顶层节点（root），且 status ≠ inbox

**滞留保护**：inbox 节点的 stale 判定**绑定 created_at（出生时间）而非 status_changed_at**——防止 Agent 通过追加无意义日志刷新时间戳来"续命"。公式：`status = 'inbox' 且 created_at < NOW() - 7 days`。一旦触发 stale，系统在 Heartbeat 时发出高优清理告警

## FR-8: 主动提醒（Heartbeat 协议）

脊髓周期性扫描图谱，将时间衰减转化为结构化事件提醒，为没有天然时间感的 Agent 补上时间维度（Rehearsal 机制）。

**触发时机**：
- **同步路径**：Session 启动时（before_prompt_build bootstrap），受严格延迟预算约束
- **异步路径**：宿主框架的 Heartbeat 轮询时；可选独立 Cron（频率可配置，默认每 15 分钟）

**扫描规则**（DRY 铁律：100% 复用 FR-5.2 风险标记引擎）：

Heartbeat 不硬编码任何时间阈值。它直接提取 FR-5.2 已计算的动态风险标记，完成风险→事件映射：

| 优先级 | 风险源 | 生成提醒类型 |
|--------|--------|-------------|
| 🔴 高 | at-risk 且 deadline < 24h | 紧急临期 |
| 🟠 高 | blocked 且被至少一个活跃节点依赖 | 关键阻塞 |
| 🟡 中 | at-risk（48h 内） | 临期提醒 |
| 🟡 中 | stale（FR-5.2 定义） | 僵尸告警 |
| 🔵 低 | inbox + stale（FR-7 created_at 定义） | 整理提醒 |

**输出格式**：Heartbeat 首先生成结构化提醒对象，再由 Bundle Assembler 渲染为**独立的系统告警区（L_Alert）**——位于 L0 看板与 L1 近景之间，时空隔离，严防无关告警混入 L1 拓扑近景导致 Agent 逻辑串线。单次 Heartbeat 最多注入 Top 3 最高优先级告警，超出部分以计数摘要呈现：

```
{ alert_type, severity, node_id, reason, suggested_action }
```

每条提醒渲染后不超过 500 tokens。

**焦点候选**：高优先级提醒（🔴🟠）生成候选焦点建议，由 FR-9 焦点调度器按优先级规则决定是否采纳。Heartbeat 自身不直接写入焦点状态。

**去重/降频**：相同 alert_type + 相同 node_id 的提醒，在该节点未发生状态变化前，不得在连续 Heartbeat 中重复推送。已推送的提醒降级为"持续监控"，仅在状态恶化时重新升级。

**约束**：Heartbeat 不修改业务事实层状态，仅生成提醒集与候选焦点建议。状态变更仍需通过 Agent 的 Tool Call 触发。

## FR-9: 焦点选择与调度

焦点（Focus）是系统维护的注意力指针，决定认知包的 L2 中心。焦点状态持久化在 session_state 中（FR-0 运行时事实状态层），为 session 级别。

**焦点驱动方式与优先级仲裁**（冲突时高优先级替换低优先级焦点槽位）：

| 优先级 | 触发类型 | 机制 |
|--------|----------|------|
| 1（最高）| 用户驱动 | 人类提到某个项目/任务 → Agent 调用 `shift_focus(node_id)` |
| 2 | 事件驱动 | 外部信息能通过确定性规则映射到唯一节点时，生成焦点候选注入 L_Alert（如消息中包含 node_id 引用）。Agent 阅读后自行决定是否调用 shift_focus。底层不自动切换焦点 |
| 3 | 时间驱动 | 来自 FR-8 Heartbeat 高优先级提醒的焦点候选（不独立重复扫描时间阈值） |
| 4（最低）| 历史残留 | 从 session_state 恢复上次 session 的焦点 |

`shift_focus` 为焦点切换请求，经焦点调度器统一裁决（节点存在性、槽位、优先级检查），非绕过调度器的强制写入。

**焦点层级**：当存在多个焦点时，系统区分主焦点（primary）与次焦点（secondary）。L2 中心围绕主焦点构建，次焦点以 L1 摘要形式存在于认知包中。

**衰减机制**：焦点超过 3 天未被触及 → 自动降级回近景（可配置）。"被触及"定义为以下任一事件：用户显式提及、Agent 对该节点发起 Tool 操作、该节点被纳入当前轮次的 L2 认知包。

**焦点切换（Saccade）**：
- 视线移开：当前焦点降级为 L1 摘要，更新 session_state 中的焦点记录
- 视线移入：新节点从 L0/L1 展开为 L2，触发认知包重建
- 切换由脊髓毫秒级执行

**无焦点模式**：若当前不存在有效焦点（无用户焦点、无自动候选、历史焦点已衰减），系统进入无焦点模式——仅推送 L0 全局背景与 L_Alert 系统告警，等待用户显式指令或新的焦点来源。

**约束**：同时活跃的焦点上限 3 个（1 主 + 2 次）。焦点队列采用 LRU（Least Recently Used）淘汰策略——当队列已满且 Agent 压入新焦点时，最久未被触及的旧焦点静默挤出，降级回 L1/L0。

## FR-10: 认知包组装（Bundle Assembly）

基于当前焦点，脊髓引擎通过图谱拓扑查询确定性组装认知包。本条定义具体组装规则，协议与分辨率模型见 FR-4。

### 组装规则与物理排列顺序

认知包在 Prompt 中的注入顺序（从上到下）：

1. **L0 背景层**：全局看板（FR-3 树形投影），~500-1k tokens
2. **L_Alert 系统告警区**：FR-8 Heartbeat 生成的结构化提醒，最多 Top 3，~500 tokens
3. **L1 近景关联层**：焦点的局部工作集，~1-3k tokens
   - 向上：parent 节点的目标摘要
   - 向下：**仅限直接子节点（Depth=1）**的未完成状态列表（严禁跨代递归展开，防 Token 爆炸）
   - 横向出边：depends_on 前置方的状态与卡点原因（谁在阻塞我）
   - 横向入边：depended_by 后置方的状态摘要（谁在等我完成——爆炸半径）
   - 平行：sibling 节点的状态摘要
4. **L2 焦点区**：主焦点的工作级上下文，~2-5k tokens
   - 焦点节点骨架全字段 + 摘要层 + 最近关键叙事（默认最近 3 天或最近 5 条，可配置）
   - 完整原始叙事仅在 Agent 通过 `expand_context(node_id, mode="history")` 显式扩包时加载
5. **跨节点旁路关联（时间近邻）**：基于 narrative_index，提取**其他节点**在最近 3 天叙事中显式提及当前焦点 node_id 的事件记录（严禁提取焦点自身叙事——L2已包含，避免数据重影）。若窗口内无外部引用，回退到最近 1 条跨节点 status_change 事件或直接省略本区块

### 多焦点处理

当存在多个焦点时，Bundle 仅围绕主焦点构建 L2；次焦点以 L1 摘要形式存在，除非 Agent 显式触发焦点切换或扩包。

### 无焦点模式

当系统处于无焦点模式时（FR-9），认知包仅包含 L0 全局背景与 L_Alert 最高优先级提醒，不组装 L1/L2。

### L1 超预算裁剪顺序

当 L1 内容超出预算时，按以下优先级保留（从高到低）：parent > depends_on（阻塞方）> depended_by（被阻塞方）> children > siblings。低优先级内容折叠为计数摘要。

### 约束

- 认知包组装严格基于事实源与确定性拓扑查询，不从派生视图反推
- v1 默认 Bundle 组装链路不依赖模糊语义搜索；如后续引入 embedding/RAG，只能作为补充召回层，不替代确定性拓扑装载

## FR-11: 受约束写入（Constrained Mutation）

Agent 对系统数据的所有变更，必须收敛为有限数量的原子化 Tool Call。Agent 不得直接读写底层存储文件。任何不在下述接口列表中的写入操作，系统一律拒绝。

### 事实层变更 Tool（v1 最小集）

改变业务事实的 Tool Call，必须留下可追溯的业务事件记录：

| Tool | 功能 | 校验 | 必传业务事件 |
|------|------|------|-------------|
| `create_node` | 创建节点（默认 status=inbox） | 必填字段检查（至少 title） | 最小初始日志（创建来源与意图，Inbox 场景可极简） |
| `update_status` | 变更节点状态 | FR-5.1 合法迁移 + 触发冒泡级联 | **结构化 reason（必传）**：actor + fact + reason |
| `update_field` | 修改节点白名单业务字段 | 字段白名单 + 类型校验 | 结构化 reason |
| `add_dependency` | 添加 depends_on 关系 | DAG 环路检测 | 结构化 reason |
| `remove_dependency` | 移除 depends_on 关系 | 关系存在性检查 | 结构化 reason |
| `attach_node` | 设置 parent_id（挂载 inbox 节点） | 目标存在性 + 防环 + FR-7 原子转正 | 结构化 reason |
| `detach_node` | 解除 parent_id（变为顶级节点） | 当前 parent_id 存在性 | 结构化 reason |
| `append_log` | 纯追加叙事（不改骨架状态） | FR-2 格式校验 + 边界检查 | —（自身即日志，传入 actor + fact + reason） |
| `unarchive` | 恢复归档节点 | FR-6 拓扑合法性 | 结构化 reason |
| `set_persistent` | 设置/取消常驻标记 | 节点存在性 | 结构化 reason |

**update_field 白名单**：仅允许修改 `summary`、`why`、`next_step`、`owner`、`deadline`、`node_type` 等业务字段。不得修改系统字段（id、created_at、status_changed_at）、推导字段（rollup_status、风险标记）或关系字段（parent_id 由 attach_node 管理、depends_on 由 add/remove_dependency 管理）。summary 写入遵循 FR-1 优先级：人工摘要 > 压缩器。

**结构化 reason 对齐 FR-2**：事实层 Tool Call 的业务事件输入为结构化字段（actor/fact/reason），非自由文本字符串。脊髓引擎使用代码模板确定性拼装落盘文本（FR-2）。

### 运行时状态 Tool

不改变业务事实，仅影响会话级运行时状态：

| Tool | 功能 | 校验 |
|------|------|------|
| `shift_focus` | 切换焦点（FR-9） | 节点存在性 + 调度器裁决 |
| `expand_context` | 请求扩展认知包（FR-4） | node_id + mode 校验 |

运行时 Tool 不要求 reason_log，不产生叙事条目。

### 写入流程与一致性

**主提交流程**：
1. Tool Call 接收 → 脊髓引擎校验（MUST）
2. 生成审计事件 events.jsonl（MUST，与 DB 同步或强制后置）
3. SQLite 事务提交（MUST）——主提交点
4. Markdown 叙事补写（MUST with repair semantics）——若失败，记录 repair event 并在后续修复，不回滚已提交的结构化状态
5. 触发级联（冒泡/风险重算）（MUST）
6. 派生视图重绘（MUST）——SQLite 内存查询极快，确保 Agent 下一轮看到最新状态，避免因过期视图导致重复 Tool Call

**幂等性约束**：所有写入类 Tool Call 应支持幂等处理或携带唯一操作 ID（operation_id），避免因重复调用导致重复叙事、重复依赖或状态抖动。

**Tool Call 返回值**：每次 Tool Call 完成后，脊髓引擎返回结构化结果（success/failure + 变更摘要 + 受影响节点列表），供 Agent 确认操作结果。

## FR-12: 叙事压缩

系统必须具备将旧叙事持续压缩为更高密度抽象层的能力，防止叙事文件无限膨胀。压缩不是删除历史，而是把历史提升为更高层次的可工作抽象。

### 压缩策略

- **规则优先**（90%+ 场景）：由确定性规则提取关键事件，压缩为摘要段落
- **LLM 辅助**（可选 fallback）：规则无法处理的复杂叙事（如长谈判链、多轮因果），交由本地 LLM（如 Ollama）生成摘要。LLM 摘要仅作为低优先级历史摘要层，不得替代结构化事实，不得覆盖人工确认的当前摘要

**规则压缩保留策略**：

| 事件类型 | 处理方式 |
|----------|----------|
| status_change | 必须保留 |
| decision | 必须保留 |
| blocker | 必须保留 |
| correction | 必须保留（含纠偏链） |
| info | 可合并或降权（不影响因果链时） |

### 触发条件

默认触发阈值（可配置）：超过 30 天**且**超过规模阈值（条目数 > 50 或 token > 2000）。时间窗与规模阈值至少满足一项。

**调度约束**：
- 作为**低优先级后台异步任务**在闲时执行（Cron 或后台进程）
- **严禁**在 Agent 交互主链路或同步 Heartbeat 中阻塞调用
- 当前焦点、近景或高风险状态的节点不优先触发压缩

**乐观并发控制（Anti-TOCTOU）**：LLM 压缩耗时较长（可达数十秒），极易与前台 Agent 写入冲突。后台 Worker 必须：读取时记录 updated_at → 离线生成摘要 → 准备落盘时再次检查 updated_at。若时间戳已变（前台刚写入），**立即丢弃本次压缩结果**，退让至下个周期重试。绝不强行覆写。

### 压缩输出

**分层写入**：压缩结果写入节点叙事文件顶部的**历史摘要区（compressed_summary）**，不覆盖节点的当前 `summary` 字段。summary 服务于 L0/L1 当前认知接口，历史摘要区服务于 L2 历史回溯。

**可追溯性**：每条压缩摘要必须带元信息——覆盖时间窗口、来源条目区间、生成方式（rule/llm）。支持追溯与必要时的重建。

**防套娃水位线**：节点维护 `last_compressed_at` 游标。每次压缩仅提取该游标之后、且满足时间阈值的增量原生叙事，绝不将旧摘要再次喂给 LLM 二次压缩。压缩完成后更新游标。

**联动 FR-2 膨胀控制**：FR-2 的"建议压缩"标记（needs_compression）触发本模块的压缩候选扫描。

**原始叙事保护与物理截断**：原始叙事不删除、不覆写。真正的"降权"发生在 FR-10 组装阶段——L2 焦点区优先加载 compressed_summary，仅在 token 预算允许时追加近期原始叙事；远古叙事（last_compressed_at 之前）不进入认知包，仅服务于审计追溯。

**压缩失败处理**：压缩失败不影响事实层与现有认知包组装。系统仅记录压缩失败事件并允许后续重试。

## FR-13: 上下文恢复流程

Agent 新 session 启动时的标准恢复流程（脊髓全自动执行，用户无感知，零 LLM 参与）：

| 步骤 | 动作 | 说明 |
|------|------|------|
| 1 | 连接事实源 | 脊髓打开 SQLite，全量加载活跃拓扑（全部非归档 nodes + edges + session_state + narrative_index）。DAG 防环与全局冒泡必须拥有完整拓扑视角，严禁局部切片加载 |
| 2 | 生成全局看板 | 投影引擎从 SQLite VIEW 生成 L0 背景视图（FR-3） |
| 3 | 执行 Heartbeat | FR-8 同步路径扫描，生成提醒集与候选焦点。受严格延迟预算约束（超时则降级跳过，异步补执行） |
| 4 | 焦点仲裁 | 综合历史残留焦点（session_state）+ Heartbeat 候选焦点，按 FR-9 优先级仲裁确定主/次焦点。校验节点存在性（已归档/不存在的焦点自动清理） |
| 5 | 组装认知包 | 按 FR-10 规则组装 L1+L2（围绕主焦点） |
| 6 | 推送 Bootstrap Context | 按 FR-10 顺序（L0 → L_Alert → L1 → L2）打包，通过 before_prompt_build hook 推入 Agent |

**冷启动预算约束**：恢复流程优先生成最小可工作 Bootstrap Context，禁止在启动阶段默认展开完整历史叙事或全量节点。

**恢复完成判定**：当 L0 背景、焦点状态、提醒集和主焦点认知包均已成功组装并推送后，当前 session 视为恢复完成。

**降级策略**（局部失效→局部降级，不阻断整轮 bootstrap）：

| 故障 | 策略 | 重建路径 |
|------|------|----------|
| 全局看板损坏 | 从 SQLite 重建 L0 | SQL VIEW 重算 |
| 焦点列表丢失/全部失效 | 无焦点模式（FR-9），仅 L0 + L_Alert | 等待用户 shift_focus |
| 单个节点叙事损坏 | 跳过该节点 L2，降级为 L1 摘要 | 不阻断其他节点恢复 |
| SQLite 损坏 | **灾难恢复模式**：从 MD 叙事文件 + events.jsonl 离线重建 | MD 解析骨架字段补 nodes 表，events.jsonl 补 edges/关系（G8）。此为最后手段，不保证实时恢复 |

恢复过程对用户透明，不需要人类主动提供背景。

# 5. 非功能需求

## NFR-1: Context 效率（对齐 FR-4 / FR-10 / 原则 P2 MSC）

认知包各层 token 预算（可配置默认值）：

| 层 | 预算 | 说明 |
|----|------|------|
| L0 全局看板 | 500 - 1k | FR-3 树形投影 |
| L_Alert 系统告警 | ≤ 500 | FR-8 Top 3 提醒 |
| L1 近景关联 | 1k - 3k | 按裁剪顺序降级（FR-10） |
| L2 焦点区 | 2k - 5k | 摘要 + 近期关键叙事 |
| **单次认知包总量** | **5k - 10k（典型）** | 复杂依赖场景可达 15-20k |

- 系统内部查询、拓扑计算和索引维护不计入认知包预算；凡实际注入 LLM 的内容，均计入上下文成本
- 遵循 MSC 原则：按任务复杂度动态调整，以上预算为默认目标区间而非硬性上限；超出时必须通过显式扩包触发
- Bootstrap Context（冷启动）≤ 典型认知包预算

## NFR-2: 维护成本

- Agent 自主维护，不需要人类手动编辑结构化文件
- 人类通过自然语言交互即可创建、更新、查询节点
- 系统文件格式人类可读（SQLite 可通过标准工具查询，Markdown 叙事可直接阅读）
- 人类可在离线状态下直接编辑 Markdown 叙事文件（仅限叙事层修正，结构化事实层仍禁止绕过 Tool Call 直接修改）。系统重启或扫描时检测并重新索引叙事层变更
- 零外部依赖：嵌入式 SQLite + 本地文件，无需安装/维护数据库服务

## NFR-3: 渐进式

- 可以从 0 个节点开始，逐步添加
- 不需要预先定义完整的项目层级
- 结构随内容自然生长（分形：从原子任务长出项目，从项目长出战略）
- Inbox 作为零摩擦入口（FR-7），任何想法可先捕获后整理

## NFR-4: 容错与可恢复性（对齐 G8 / FR-13）

- **节点隔离**：单个节点数据损坏不影响其他节点，不阻断全局启动（FR-13 局部降级）
- **孤儿节点**：缺少 parent_id 的节点仅在被明确声明为顶层时视为合法 root；否则标记为结构异常，在 L0 看板中暴露为待修复项
- **状态一致性**：rollup_status 始终以子节点事实重新计算（FR-5），不缓存不信任旧值；该派生结果不得直接覆盖父节点人工维护的主状态
- **灾难恢复**：SQLite 损坏时可从 MD 叙事 + events.jsonl 离线重建（FR-13，最后手段）
- **压缩安全**：压缩失败不影响主系统（FR-12），原始叙事永不删除

## NFR-5: 性能

- **Tool Call 响应**：目标 < 100ms（校验 + DB写入 + 级联 + 视图重绘，不含异步 repair/压缩）
- **认知包组装**：目标 < 50ms（常规焦点规模下）
- **冷启动恢复**：目标 < 1s；若派生索引缺失，可降级到秒级重建模式
- **Heartbeat 同步扫描**：目标 < 50ms；超预算时仅扫描高优先级集合
- **节点规模**：v1 目标支持 1000–5000 活跃节点（SQLite 嵌入式）；更大规模通过索引优化和分层装载扩展

## NFR-6: 安全与数据主权

- **写入权限**：所有业务事实变更必须通过 FR-11 Tool Call 接口，无旁门
- **数据本地化**：全部数据存储在本地文件系统，不依赖云服务，不外传
- **审计可追溯**：所有状态变更留有 events.jsonl 审计记录 + Markdown 叙事
- **LLM 隔离**：LLM 只读认知包、只通过 Tool Call 写入；无法直接操作文件系统
- **人类最终控制权**：人类可通过管理接口修改结构化事实层，并可直接修正 Markdown 叙事层；任何绕过 Tool Call 的修改均被系统视为外部变更并重新索引

## NFR-7: 可观测性

- **events.jsonl**：所有 Tool Call 操作的结构化审计日志
- **repair 记录**：MD 补写失败等异常的修复追踪
- **启动诊断**：冷启动过程中的节点解析异常以 L0 告警形式暴露给 Agent
- **压缩追溯**：每条压缩摘要带元信息（时间窗口、来源、生成方式）
- **上下文预算观测**：Bundle token 估算、L0/L1/L2 各层占比、扩包触发次数

# 6. 约束

- **存储约束**：结构化事实层使用嵌入式 SQLite，叙事层使用 Markdown 文件；v1 不依赖任何外部数据库服务、远程状态存储或托管型数据基础设施。

- **运行时约束**：Agent 在 OpenClaw 框架内运行。所有事实层写操作（状态变更、关系变更、字段更新、叙事追加）必须通过 Tool Call 调用脊髓引擎（spine.py）执行，Agent 不得直接修改存储层。

- **兼容性约束**：v1 不修改现有 `MEMORY.md` 与 `memory/` 日志系统。旧系统仅作为只读历史输入源，可被受约束检索或引用，但不得被新系统写入、覆盖或删除。

- **数据安全约束**：任何对历史记忆材料（包括 `MEMORY.md`、`memory/`、叙事文件及审计日志）的删除、覆盖、截断或不可逆修改，都必须经过用户明确确认。默认情况下，系统仅允许追加、读取和可恢复的标记操作。

- **写入边界约束**：除受约束 Tool Call 和明确声明的系统维护流程（如 FR-12 后台压缩）外，不得存在任何旁路机制修改事实层数据。

- **Token 预算约束**：项目管理系统单次注入（含 Context Bundle + Tool 定义 schema）默认目标不超过 context window 的 5%，并受 NFR-1 绝对预算上限约束。复杂依赖场景允许弹性至 10%，但必须通过显式扩包触发，且可观测、可解释。
