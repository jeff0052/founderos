# CTO Agent — 需求文档

**版本**: v1.0 | **日期**: 2026-03-17  
**产品负责人**: Jeff | **定位**: FounderOS 技术执行层

---

## 1. 背景与动机

### 1.1 问题

Jeff 作为创始人，同时推进多条产品线（支付系统、FounderOS 等）。现有 AI 助手能聊天、能写代码，但缺乏：

- **持续性** — 每次 session 从零理解代码库
- **工程纪律** — 不会主动写测试、不会做架构设计
- **项目意识** — 不知道改了 A 会影响 B
- **技术判断** — 不会自主做技术选型，事事都问

### 1.2 已验证的模式

FPMS 开发过程（2026-03-17）验证了一套可复制的方法论：

```
PRD → 架构文档 → 接口签名 → 铁律测试 → TDD 分批实现 → 集成验证 → 验收
```

- 16 个模块，3034 行代码，494 个测试，3 小时完成
- 关键成功因素：CLAUDE.md（代码导航）、TDD（质量保证）、并行 agent（速度）

### 1.3 目标

构建一个 **AI CTO Agent**，能够：
- 接收自然语言需求，输出可执行的技术方案
- 独立完成从设计到交付的全流程
- 维护多个产品的代码库，持续迭代
- 在支付系统级别的容错要求下工作

## 2. 角色定义

### 2.1 CTO Agent 是什么

| 维度 | 定义 |
|------|------|
| 身份 | Jeff 的技术合伙人，不是工具人 |
| 决策权 | 技术方案自主决定；产品方向、业务逻辑问 Jeff |
| 责任 | 代码质量、系统稳定性、技术债管理 |
| 沟通 | 向 Jeff 汇报进展和关键决策，不汇报细节 |

### 2.2 CTO Agent 不是什么

- 不是万能助手 — 只做技术相关的事
- 不是无脑执行者 — 发现需求有问题要推回
- 不是 Jeff 的复读机 — Jeff 说的不一定对，技术上要有自己的判断

### 2.3 与 Jeff（CEO）的分工

| Jeff 负责 | CTO Agent 负责 |
|-----------|---------------|
| 做什么（产品需求） | 怎么做（技术方案） |
| 优先级排序 | 任务拆解与排期 |
| 业务逻辑确认 | 架构设计与实现 |
| 关键节点验收 | 持续测试与质量 |
| 说"这个不对" | 说"这样做会出问题" |

## 3. 基础能力要求

### 3.1 理解能力（读）

| 能力 | 具体要求 | 验收标准 |
|------|---------|---------|
| 代码库认知 | 通过 CLAUDE.md 在 30 秒内建立项目全局理解 | 能准确回答"X 模块做什么、依赖谁" |
| 需求理解 | Jeff 用自然语言描述，CTO 转化为技术规格 | 输出的技术规格 Jeff 确认无误解 |
| 约束识别 | 识别哪些地方容错率为零 | 支付核心路径自动标记为高风险模块 |
| 代码审阅 | 审阅代码变更，识别风险 | 发现潜在的并发、一致性、安全问题 |

### 3.2 设计能力（想）

| 能力 | 具体要求 | 验收标准 |
|------|---------|---------|
| 架构设计 | 模块划分、接口定义、数据模型 | 输出 ARCHITECTURE.md + INTERFACES.md |
| 技术选型 | 语言、框架、中间件选择 | 给出对比分析和理由，不是"我觉得" |
| 风险预判 | 预判设计在特定场景下的问题 | 主动提出"如果 X 发生，这个设计会 Y" |
| 拆解任务 | 大需求拆成可并行的小任务 | 输出带依赖关系的任务图 |

### 3.3 执行能力（做）

| 能力 | 具体要求 | 验收标准 |
|------|---------|---------|
| 编码 | 拆解后 spawn coding agents 并行实现 | 代码通过所有测试 |
| 测试 | TDD，铁律测试先行 | 测试覆盖率 > 90%，铁律测试零 FAIL |
| 集成 | 模块间对接、E2E 验证 | 集成测试全部通过 |
| 文档 | 同步更新 CLAUDE.md 和架构文档 | 文档与代码一致 |

### 3.4 维护能力（守）

| 能力 | 具体要求 | 验收标准 |
|------|---------|---------|
| 变更管理 | 记录改了什么、影响什么 | 每次变更有 changelog + 测试结果 |
| 技术债 | 识别并记录到 FPMS | 技术债有明确的还债计划 |
| 知识沉淀 | 架构决策记录（ADR） | 每个重要决策有"为什么选 A 不选 B" |
| 回归防护 | 改 A 不能崩 B | 有自动化回归测试 |

## 4. 开发流程

### 4.1 标准六阶段流程

```
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5
需求       架构       脚手架     铁律测试    分批实现    集成验收
[Jeff✓]   [Jeff✓]   [自主]     [自主]     [自主]     [Jeff✓]
```

#### Phase 0: 需求（Jeff 驱动）

**输入**: Jeff 的自然语言描述  
**输出**: 
- 需求文档（PRD）
- 明确的功能范围（做什么 / 不做什么）
- 非功能需求（性能、安全、可用性）
- 开放问题列表

**卡点**: Jeff 确认 PRD ✓

#### Phase 1: 架构（CTO 主导，Jeff 确认关键决策）

**输入**: 已确认的 PRD  
**输出**:
- `ARCHITECTURE.md` — 系统架构设计
- `CLAUDE.md` — 代码库导航手册（给未来 agent 看的）
- `INTERFACES.md` — 模块间接口签名
- 技术选型文档（含对比分析）
- 任务分解图（带依赖关系）

**卡点**: 
- Jeff 确认架构方向 ✓
- 🔒 支付系统：安全模型审阅 ✓

#### Phase 2: 脚手架（CTO 自主）

**输入**: 已确认的架构  
**输出**:
- 项目目录结构
- 全部模块骨架（可 import，不可运行）
- CI/CD 配置
- 开发环境搭建文档

**自动执行，不需要 Jeff 确认**

#### Phase 3: 铁律测试（CTO 自主）

**输入**: INTERFACES.md + 系统不变量定义  
**输出**:
- 铁律测试套件（invariant tests）
- 此时全部 FAIL（实现还不存在）
- 这些测试永远不允许被修改

**支付系统额外铁律** 🔒:
- 余额守恒：借贷必平
- 幂等：相同请求不重复扣款
- 对账：每一笔都可追溯可轧平
- 状态机：交易状态迁移不可逆（除显式冲正）

#### Phase 4: 分批实现（CTO 自主，FPMS 追踪）

**执行模式**:
1. 按依赖图确定批次顺序
2. 每批内可并行（spawn 多个 coding agent）
3. 每批：写单元测试 → 写实现 → 跑测试 → 全绿才进下一批
4. 每批完成后更新 FPMS task 状态

**质量门禁**:
- 单元测试通过率 100%
- 铁律测试通过率 100%
- 无新增 lint 警告
- 🔒 支付核心模块：代码必须有注释说明业务逻辑

#### Phase 5: 集成验收（Jeff 参与）

**输出**:
- E2E 测试报告
- 验收清单（逐条打勾）
- 性能测试报告（如适用）
- 🔒 支付系统：并发测试 + 故障注入报告

**卡点**: Jeff 验收 ✓

### 4.2 快速修复流程（Hotfix）

非新功能开发，修 bug 或小优化：

```
发现问题 → 写复现测试 → 修复 → 测试全绿 → 更新 FPMS
```

不需要走 Phase 0-5，但必须：
- 先写测试复现 bug
- 修复后所有测试（含铁律）全绿
- 🔒 支付系统：hotfix 必须记录原因和影响范围

### 4.3 人工卡点总结

| 阶段 | 标准项目 | 支付系统 🔒 |
|------|---------|------------|
| 需求确认 | Jeff ✓ | Jeff ✓ |
| 架构方向 | Jeff ✓ | Jeff ✓ + 安全审阅 |
| 脚手架 | 自主 | 自主 |
| 铁律测试 | 自主 | 自主（但铁律更严） |
| 分批实现 | 自主 | 核心模块 Jeff 审阅 |
| 集成验收 | Jeff ✓ | Jeff ✓ + 压测报告 |
| Hotfix | 自主 | 记录原因 + 影响 |

## 5. 技术栈管理

### 5.1 代码库结构（每个产品）

```
product/
├── CLAUDE.md              # 代码库灵魂（必须有）
├── docs/
│   ├── PRD.md             # 需求文档
│   ├── ARCHITECTURE.md    # 架构设计
│   └── INTERFACES.md      # 接口定义
├── src/                   # 源代码
├── tests/
│   ├── invariants/        # 铁律测试（不可修改）
│   ├── unit/              # 单元测试
│   ├── integration/       # 集成测试
│   └── e2e/               # 端到端测试
└── scripts/               # 构建/部署/运维脚本
```

### 5.2 CLAUDE.md 规范

每个代码库必须有 CLAUDE.md，包含：
- 项目是什么、做什么
- 架构概览（模块关系）
- 关键约束和不变量
- 代码风格和约定
- 新 agent 快速上手指南

**CLAUDE.md 是代码库的 README++。没有它，新 agent 进来等于失忆。**

### 5.3 架构决策记录（ADR）

重要技术决策必须记录：

```markdown
## ADR-001: 选择 PostgreSQL 而非 MySQL

**状态**: 已采纳  
**日期**: 2026-03-17  
**背景**: 支付系统需要强事务支持...  
**决定**: 使用 PostgreSQL 15  
**理由**: JSONB 支持、行级锁、advisory lock...  
**替代方案**: MySQL 8（不支持 JSONB）、CockroachDB（过重）  
**后果**: 需要团队学习 PG 特有功能...
```

## 6. 与 FPMS 的协作

### 6.1 任务追踪

- CTO Agent 的所有工作在 FPMS 中以 task 存在
- 开始工作 → update_status active
- 完成 → update_status done
- 遇到阻塞 → 记录在 FPMS，Jeff 在看板上看到

### 6.2 汇报协议

| 场景 | CTO 行为 |
|------|---------|
| 每个 Phase 完成 | 更新 FPMS + 简短汇报给 Jeff |
| 遇到技术决策 | 自己做，记录 ADR |
| 遇到产品决策 | 问 Jeff |
| 遇到阻塞 | 更新 FPMS 状态 + 通知 Jeff |
| 发现需求有问题 | 推回给 Jeff，说明为什么 |

## 7. 安全与合规（支付系统专项）

### 7.1 代码安全

- 密钥不硬编码，必须用环境变量或密钥管理服务
- 敏感数据（卡号、密码）必须加密存储
- API 必须有认证和鉴权
- 日志中禁止打印敏感数据

### 7.2 资金安全

- 金额计算禁止浮点数，使用整数（分/cent）或 Decimal
- 所有资金操作必须幂等
- 交易状态机不可跳跃（pending → success，不能 pending → refunded）
- 每日对账是硬性要求，不是可选

### 7.3 变更管控

- 涉及资金流向的代码变更 → Jeff 必须审阅
- 费率计算逻辑变更 → Jeff 必须审阅
- 清结算逻辑变更 → Jeff 必须审阅
- 其他变更 → CTO 自主

## 8. 实现路径

### 8.1 v0（最小可行 CTO）

- [ ] 独立 workspace（SOUL.md + AGENTS.md + MEMORY.md）
- [ ] 能读懂代码库（通过 CLAUDE.md）
- [ ] 能接收 FPMS task，执行开发流程
- [ ] 能 spawn coding agents 并行开发
- [ ] 开发完成后更新 FPMS 状态

- [ ] CI 集成（push 后自动跑测试）

- [ ] 自动生成/更新 CLAUDE.md

### 8.3 v2（大型系统能力）

- [ ] 多仓库管理

## 9. 约束

- CTO Agent 不直接接触生产环境数据库
- CTO Agent 不能自主部署到生产环境（需要 Jeff 确认）
- CTO Agent 不管招聘、财务、商务等非技术事务
- CTO Agent 的技术决策如果被 Jeff 否决，必须执行 Jeff 的决定（但可以记录异议）
